# Elegant READMEs
- **Source**: https://www.yegor256.com/2019/04/23/elegant-readme.html
- **Retrieved**: 2026-05-15
- **Author**: Yegor Bugayenko

## Structure and Organization (Prescribed Order)

1. **Logo and Badges** — Start with a logo (max 100px height) followed by badges
2. **Description** — One paragraph explaining what the product is
3. **How to Use It** — Quick-start instructions for beginners
4. **Use Cases** — The main body with specific examples
5. **How to Contribute** — Instructions for potential developers
6. **Releases, License, Changelog** — Reference existing GitHub features instead of duplicating

## Key Formatting Rules

Bugayenko emphasizes treating README as source code:

- "Eighty characters. That's it." — Maintain strict line width except for badges
- Use single empty lines between sections
- Never indent text; always start from the left margin
- Limit badges to five per line with consistent heights
- Use triple-backticks for code samples

## Critical Content Advice

**On Description:** Keep it to "one single paragraph. No bullets here, no new lines, no indentations."

**On How-To Section:** Address complete newcomers with immediate actionable steps. The author notes he has reader attention for "another 60 seconds," so urgency matters.

**On Use Cases:** Avoid replacing autogenerated documentation; keep content high-level and redirect to other resources. He warns: "Don't focus on yourself, we don't care about you."

## What to Avoid

- Repeating information already in GitHub's metadata
- Over-explaining frameworks or tools ("Don't teach us")
- Verbose prose or philosophy
- Changelogs (use releases tab instead)
- Contributor lists (GitHub provides this)
- Headers beyond level two (`##`)

## Tone and Audience

Bugayenko advocates addressing "a junior developer" and keeping language minimal. He states: "Nobody is interested in reading your prose for more than a few seconds."
