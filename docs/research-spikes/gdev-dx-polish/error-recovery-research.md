# Error Recovery and Self-Healing Research

## Research Question

What happens when gdev-managed environments break? Should gdev include a `repair`/`fix` command, and what common failure modes should it handle?

## The "Doctor" Pattern: Prior Art

The best-established pattern for CLI diagnostic tools is the "doctor" command, exemplified by:

**flutter doctor**: Scans for Flutter SDK, Android toolchain, Xcode, Chrome, connected devices. Shows check/cross marks with actionable fix instructions. `-v` flag for verbose output. Run after install, upgrades, and when encountering build issues.

**brew doctor**: Checks for common Homebrew installation problems -- unlinked kegs, outdated Xcode CLT, permission issues, config file conflicts.

**npm doctor**: Validates npm installation, registry connectivity, cache integrity, permissions.

**The pattern**: Diagnose -> Report -> Suggest fixes (sometimes auto-fix).

gdev already plans a `gdev devenv doctor` command (Phase 9). This research focuses on what it should check and whether auto-repair is feasible.

## Common Failure Modes in gdev-Managed Environments

### Category 1: Nix/devenv Failures

| Failure | Cause | Detection | Recovery |
|---------|-------|-----------|----------|
| `devenv shell` fails to evaluate | Syntax error in devenv.nix, broken flake input | `devenv info` exit code | Show error, offer `gdev init --repair` to regenerate machine-owned files |
| Nix store corruption | Interrupted GC, disk full, power loss | `nix store verify --all` | `nix store repair --all` (slow but safe) |
| Stale devenv lock | devenv.yaml changed but `devenv update` not run | Compare devenv.yaml mtime vs devenv.lock mtime | Run `devenv update` |
| Missing Nix binary cache | Cache server down, network issue | Attempt cache fetch | Fall back to local build, warn about latency |
| devenv version mismatch | Team on devenv 2.x, local install is 1.x | `devenv version` check | Warn, suggest upgrade path |

### Category 2: Generated Config Corruption

| Failure | Cause | Detection | Recovery |
|---------|-------|-----------|----------|
| Machine-owned file manually edited | Developer modified .envrc, devenv.yaml | SHA256 hash mismatch (gdev already tracks this) | Warn, offer regenerate with `--force` |
| settings.json merge conflict | Three-way merge failure after update | JSON parse failure or schema validation | Keep backup, regenerate from template, apply user customizations |
| CLAUDE.md section markers damaged | Manual editing removed markers | Regex check for section boundaries | Regenerate section markers, preserve non-marker content |
| Pre-commit hooks not installed | `.git/hooks/` missing or outdated | Check hook files exist and match expected content | Reinstall via `prek install` or equivalent |
| .mcp.json invalid | Manual editing, missing commas | JSON parse | Regenerate from tool registry |

### Category 3: Tool/Package Failures

| Failure | Cause | Detection | Recovery |
|---------|-------|-----------|----------|
| Security tool missing | Not installed, PATH issue | `which <tool>` check | Suggest `gdev enable <tool>` or check devenv packages |
| Hook script error | Python version mismatch, missing deps | Hook exit code != 0 on test run | Check Python version, reinstall deps |
| Lock file invalid | Corrupted by merge conflict, manual edit | Per-ecosystem validation (npm: `npm ci --dry-run`, pip: parse syntax) | Delete and regenerate from manifest |
| Registry proxy unreachable | Network, config, proxy down | HTTP health check | Fall back to direct registry, warn about security gap |

### Category 4: Environment Drift

| Failure | Cause | Detection | Recovery |
|---------|-------|-----------|----------|
| gdev version newer than config | gdev updated, configs generated by older version | Version stamp in `.gdev.yaml` vs `gdev version` | Offer `gdev init --update` |
| New ecosystem detected | Developer added package.json to a Go project | Detection engine finds unmanaged ecosystem | Suggest `gdev init` to add ecosystem |
| Tool removed from gdev | gdev update drops deprecated tool | Config references non-existent tool | Warn, offer migration path |

## Design: `gdev devenv doctor` + `gdev repair`

### `gdev devenv doctor` (Diagnostic Only)

Reports environment health without modifying anything. Follows flutter doctor's UX pattern:

```
$ gdev devenv doctor
gdev Doctor
===========

[✓] gdev v1.2.0 (config version: 1.2.0)
[✓] devenv v2.1.0 (shell evaluates successfully)
[✓] Nix store (healthy, 12.3 GB used)
[!] Pre-commit hooks (installed but outdated — run gdev repair)
[✓] Claude Code settings (valid JSON, 48 deny rules active)
[✗] .envrc (modified from generated version — run gdev repair --file .envrc)
[✓] Security tools (6/6 enabled: prek, gitleaks, semgrep, osv-scanner, cosign, scancode)
[!] devenv.lock (3 inputs have updates available)
[✓] Branch naming hook (active, pattern: ^(feat|fix|chore|docs|refactor|test|ci)/[a-z0-9-]+$)

Issues found: 2 warnings, 1 error
Run gdev repair to fix automatically, or gdev devenv doctor -v for details.
```

**Key checks** (ordered by importance):
1. gdev version vs config version compatibility
2. devenv.nix evaluates without error
3. Nix store health (quick check, not full verify)
4. Generated file integrity (SHA256 hash comparison)
5. Hook installation status
6. Security tool availability
7. Lock file validity
8. Environment variable completeness

### `gdev repair` (Auto-Fix)

Fixes what can be safely fixed automatically:

```
$ gdev repair
Repairing gdev environment...

[fixed] Pre-commit hooks reinstalled
[fixed] .envrc regenerated (backup saved to .envrc.bak)
[skip]  devenv.lock — run 'devenv update' manually (may change dependencies)
[skip]  settings.json — has user customizations, use 'gdev repair --force' to regenerate

Repaired 2 issues. 2 issues require manual action.
```

**Auto-fix rules**:
- Machine-owned files (no user edits detected): regenerate
- Machine-owned files WITH user edits: backup + regenerate only with `--force`
- Hooks: always safe to reinstall
- Lock files: never auto-update (may change dependency versions)
- devenv.nix: NEVER auto-modify (the plan already establishes this principle)

### `gdev repair --file <path>` (Targeted Fix)

Fix a specific file. Useful when doctor reports one issue and the developer wants surgical repair.

## Key Design Principles

1. **doctor is read-only, repair is write**. Never mix diagnosis and modification in one command.
2. **repair is conservative by default**. Only fix what's unambiguously safe. `--force` for aggressive repair.
3. **Always backup before overwriting**. `<file>.bak` with timestamp.
4. **Hash tracking is the foundation**. The SHA256 hash system from the migration strategy design enables all corruption detection. No hash = can't detect drift.
5. **Exit codes matter**. `gdev devenv doctor` should exit 0 (healthy), 1 (warnings), 2 (errors) for CI integration.

## Comparison: gdev repair vs "Just Re-run gdev init"

| Aspect | `gdev repair` | `gdev init` (re-run) |
|--------|---------------|---------------------|
| Scope | Only broken/drifted files | All files |
| User edits | Preserves by default | Merge strategy applies |
| Speed | Fast (targeted) | Slower (full evaluation) |
| Safety | Very safe (backup + conservative) | Safe (hash tracking) but more disruptive |
| Use case | "Something broke, fix it" | "I want the latest defaults everywhere" |

Both are needed. `gdev repair` is for "fix what's broken." `gdev init --update` is for "bring everything current."
