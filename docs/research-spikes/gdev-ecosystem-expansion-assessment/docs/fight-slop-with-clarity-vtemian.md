# Fight Slop with Clarity
- **Source**: https://blog.vtemian.com/post/fight-slop-with-clarity/
- **Retrieved**: 2026-05-14

## Definition of "Slop" and Problem Description

**Core Definition:**
"Slop isn't what AI produces. Slop is what any multiplier produces when there's no clarity behind it." The author positions slop as analogous to email spam—the medium isn't inherently the problem; rather, the absence of intentional direction creates degraded output.

**The Core Problem:**
AI functions as a multiplier that scales whatever input it receives. Without foundational clarity, even technically correct work becomes noise at scale. The author illustrates this through the "Productivity Gap": experienced developers using AI felt 20% faster despite being 19% slower in actual task completion—a forty-point perception-reality gap.

**Historical Parallel:**
The post traces slop creation predating AI through organizational procedures. When measures become targets rather than descriptive tools (Goodhart's Law), documentation divorces from actual practice, creating "fog dressed up as control."

## Recommended Techniques and Practices

**The Core Questions (Infrastructure Approach):**
Rather than writing-focused clarity, the author emphasizes answering unsexy foundational questions:
- What problem am I solving?
- Whose problem is it?
- What does concrete success look like?
- What am I willing to exclude to maintain clarity?

**The Tarpit Test:**
"If a tool sells itself as a replacement for thinking clearly, it's a tarpit." Conversely, tools that amplify existing clear thinking produce value. This operational test applies to any tool across domains.

**Engineering Evolution:**
The author references movement from "vibe coding" (Andrej Karpathy's term for exploratory, pattern-matching development) toward systematic, intentional approaches. For production work, clarity cannot be outsourced—thinking precedes execution.

## Philosophy on Code Quality and Engineering Discipline

**Clarity as Non-Degrading Input:**
"Clarity is the only input that, when multiplied at industrial speed, doesn't degrade."

**Kentaro Toyama's Law of Amplification:**
Technology amplifies existing human forces—"ChatGPT helps honest writers brainstorm and helps bad students cheat." The multiplier itself is neutral; surrounding forces determine outcomes.

**The Spaghetti Principle:**
Bolting new capabilities onto unclear foundations creates exponential complexity rather than solutions. True fixes require architectural clarity, not technological upgrades. Higher technology tiers make unclarity more expensive, not more manageable.

## Specific Technical Recommendations

No specific prompts, configurations, or technical workflows are detailed. The guidance remains strategic rather than tactical:

- Prioritize clarity questions before tool selection
- Recognize when tools replace thinking (negative signal) versus amplify existing direction (positive signal)
- Avoid using AI as substitution for foundational problem definition

## Concrete Examples and Actionable Advice

**The Factorio Analogy:**
Describes how bolting technological advancement onto spaghetti architecture creates cascading inefficiency. The solution isn't better belts—it's redesigning from clarity about core requirements.

**The Procedure Stack Problem:**
Organizations create procedures to address edge cases, which generate new procedures for new gaps, until documentation describes pretense rather than practice. The fix targets clarity about what to protect, not procedure multiplication.

**The Only Viable Input:**
"Without clarity, AI is the most expensive way ever invented to be busy." This frames the core actionable principle: clarity precedes tool selection, not vice versa.
