# Research Summary: Devenv.sh Security Boilerplate

## Overview
Deep investigation into devenv.sh: how it works internally, its relationship to Nix, and how to configure a boilerplate default setup that provides developers with invisible-by-default security. Focus areas include package integrity verification, supply chain attack prevention, sandboxing of build/install scripts, protection against compromised packages being installed or executed, and defense-in-depth strategies that work without developer friction.

## Topics

### Architecture & Internals (deep)
Devenv.sh is a Rust CLI (53.9%) + NixOS module system (44.3% Nix) that provides a high-level declarative interface over Nix for developer environments. Its architecture has three layers: `devenv.yaml` controls what gets evaluated (inputs, imports, nixpkgs configuration), `devenv.nix` defines the environment through NixOS module options (packages, languages, services, processes, hooks), and the Rust CLI orchestrates evaluation, caching, and process management. Version 2.0 fundamentally changed the evaluation architecture by replacing multiple `nix` CLI process invocations with direct Nix C API calls via nix-bindings-rust FFI, enabling per-attribute incremental caching with sub-100ms activation when nothing changed. The default package source is `devenv-nixpkgs/rolling`, a Cachix-maintained fork of nixpkgs-unstable with additional patches and less community review than upstream. `devenv.lock` uses the Nix flake lock format (schema version 7) pinning input commits and NAR hashes but not individual packages. There is no separate plugin API -- all extensibility uses NixOS modules (50+ languages, 40+ services). Binary cache trust relies on Ed25519 public key verification with `devenv.cachix.org` auto-trusted by default. Runtime is completely unsandboxed (sandbox support only exists as a draft PR using bubblewrap). Devenv requires impure evaluation (cannot use Nix's pure eval mode), which is a key architectural difference from vanilla `nix develop`. The `clean.enabled` shell option and `impure` mode flag are the primary isolation controls. Comparison with alternatives shows devenv offers the lowest barrier to entry and most integrated solution but introduces vendor dependency on Cachix and cannot achieve the evaluation purity of raw flake-based shells. See [detailed report](architecture-research.md).

### Prior Art & Community Practices (deep)
No security-hardened devenv.sh boilerplate exists in the community. The ecosystem is fragmented: individual security capabilities (build sandboxing, signature verification, vulnerability scanning, secrets management) exist but are not assembled into a cohesive template. devenv.sh is actively developing security features -- SecretSpec (production since v1.8) for declarative secrets, and two competing sandbox PRs (bubblewrap full-shell vs Landlock per-executable, both draft). The broader Nix ecosystem has maturing supply chain infrastructure: the nixpkgs security tracker became operational in 2025, SBOM tooling (sbomnix) is functional, and vulnix provides Nix-native CVE scanning. Three real-world incidents inform the threat model: a GitHub Actions vulnerability that could have compromised all of nixpkgs, documented cache poisoning attack vectors, and nixConfig injection risks. Comparison with devcontainers/Codespaces shows Nix excels at reproducibility and auditability but lacks runtime isolation -- the devcontainer generator bridges this gap. A hardened boilerplate should include 6 immediately actionable measures (input pinning, SecretSpec, security pre-commit hooks, cache hygiene, vulnix scanning, trust model docs), 4 near-term measures (SBOM generation, sandbox enablement, devcontainer layer, CA derivations), and 3 systemic improvements awaiting ecosystem maturity. See [detailed report](prior-art-research.md).

### Configuration Options Inventory (deep)
Complete survey of all devenv.sh configuration knobs relevant to security across three layers: devenv.nix module options (13 groups covering packages, env, enterShell, enterTest, scripts, dotenv, git-hooks, processes, containers, files, services, overlays, unsetEnvVars), devenv.yaml settings (8 groups covering inputs, nixpkgs license/package controls, clean environment, impure mode, imports, secretspec, require_version, reload), and Nix daemon settings (12 security-critical options including sandbox, restrict-eval, require-sigs, trusted-substituters, trusted-users). Key findings: (1) Almost all security features are opt-in -- defaults are permissive. (2) `clean.enabled` is the most impactful single toggle, preventing ambient credential leakage from parent shells. (3) Only one built-in pre-commit hook (`ripsecrets`) targets security scanning; `gitleaks`, `trufflehog`, `semgrep` require custom hook definitions. (4) devenv.lock pins input sources (nixpkgs commits), not individual packages -- no CVE scanning of locked inputs. (5) Neither processes nor containers provide runtime isolation -- processes share user/env/filesystem with no namespacing. (6) SecretSpec with runtime loading (`secretspec run`) is the only mechanism that keeps secrets out of the shell environment. (7) `devenv.local.nix` can override all security controls without team visibility. (8) Nix daemon settings (sandbox, require-sigs, trusted-users) cannot be set from devenv.nix -- they require system-level configuration. Report includes a complete hardened boilerplate example. See [detailed report](config-options-research.md).

### Nix Security Mechanisms (deep)
Nix provides a layered security model -- cryptographic store integrity, build-time sandboxing via Linux namespaces, evaluation-time purity constraints, and a daemon-based trust model -- but devenv.sh inherits these without enhancing them. Nine mechanisms were analyzed in depth with devenv interaction mapped for each. The build sandbox (PID/mount/network/IPC namespace isolation) is on by default on Linux but has suffered 3 historical escape CVEs and does not apply to developer shell sessions. Binary cache signature verification (Ed25519, require-sigs=true) works well but devenv adds 2 Cachix caches as additional trust surfaces, and its documentation recommends adding users to trusted-users, which is explicitly root-equivalent per Nix's own docs. Pure evaluation mode is enabled by default with flakes, providing meaningful protection against eval-time side channels, but devenv supports disabling it via `impure: true`. Flake lock files provide strong SHA-256 narHash integrity verification, but `nix flake update` is the real attack surface. CA derivations could eliminate cache signing trust requirements but remain experimental (~65% to stabilization). Store path integrity relies on immutable paths and NAR hash verification but has no runtime monitoring. NixOS-specific features (AppArmor, SELinux) are not properly integrated. The single most impactful hardening action is replacing the dangerous trusted-users recommendation with system-level trusted-substituters configuration. See [detailed report](nix-security-mechanisms-research.md).

### Security Attack Surface & Threat Model (deep)
Complete threat model mapping 10 major attack vectors with 25+ sub-vectors across the full devenv.sh stack. Vectors cover: package source attacks (malicious overlays, nixpkgs compromise, flake input typosquatting), binary cache poisoning (key compromise, misconfiguration, provenance gaps in unsigned Deriver field, all-or-nothing third-party cache trust), shell hook injection (enterShell, scripts, git hooks, re-evaluation fork bombs), module/plugin system trust boundaries (no capability model, evaluation unsandboxed), direnv integration risks (devenv.nix changes bypass direnv's hash-check, TOCTOU on .envrc, silent re-evaluation on git pull), build-time code execution (FOD network access, setup hooks, post-build hooks), flake input manipulation (lock tampering, follows substitution, registry attacks, --override-input CLI bypass), environment variable leakage (world-readable Nix store, historical env-vars advisory, impureEnvVars exfiltration, .env file risks), process/service management (no namespace isolation, no capability dropping, port-squatting attacks), and devenv's own supply chain (curl|sh Nix install, auto-trusted devenv.cachix.org, bundled nixpkgs fork). Three cross-cutting architectural observations: (1) Nix evaluation is unsandboxed with the calling user's full privileges -- the single most important security fact; (2) devenv's trust model is "trust on first allow" with no per-change approval for devenv.nix modifications; (3) all defenses converge on code review as the primary and often only control point. See [detailed report](security-surface-research.md).

### Trust Model for Developers (deep)
Developer-facing document enumerating every trust dependency activated by `devenv shell` -- the Nix daemon, devenv binary, nixpkgs (including the devenv-nixpkgs fork issue), binary caches (3 by default, single-key Ed25519 signing, all-or-nothing trust), flake inputs (unsandboxed Nix evaluation with full user privileges), devenv.nix (code review as the sole security boundary), devenv modules (unrestricted NixOS modules with no capability model), and direnv (hash-checks .envrc but not devenv.nix changes). Includes a verification matrix mapping each dependency to its verification mechanism, missing verification, and attacker prerequisites. Provides a concrete code review checklist covering devenv.nix (enterShell/scripts/overlays/hooks/cachix changes), devenv.yaml (new inputs, impure mode, clean env, nixpkgs source changes), devenv.lock (input owner/repo changes, unexpected new nodes), and .envrc (anything beyond `use devenv`). Red flags organized in three tiers: block-PR (new binary caches, impure:true, fetchurl in .nix, overlays from external sources), elevated-scrutiny (scripts shadowing common tool names, custom git hook entries, capability grants, dotenv enablement), and information-only (normal lock updates, package additions, module enablement). Maps each hardened boilerplate choice to the specific threat it mitigates. Key message: Nix evaluation is unsandboxed and code review of devenv files is the primary (often only) security boundary. See [detailed report](trust-model-research.md).

### System-Level nix.conf Hardening (deep)
Comprehensive guide to the Nix daemon settings that devenv.sh cannot control but which form the foundation of devenv security. Covers 10 settings in depth with mechanism, default, hardened value, what breaks, and how to handle breakage for each. The three most critical changes: (1) `trusted-users = root` (removing all other users) -- devenv's recommendation to add users here is root-equivalent per Nix's own docs, replaced by `trusted-substituters` for scoped cache access; (2) `sandbox-fallback = false` -- prevents silent fallback to unsandboxed builds when the kernel lacks namespace support; (3) `connect-timeout = 10` + `download-attempts = 3` -- limits exposure to slow or malicious caches that could stall builds indefinitely. Additional settings: `require-sigs = true` (explicit defense against accidental override), `filter-syscalls = true` (seccomp for setuid/xattr prevention), `restrict-eval` + `allowed-uris` (recommended for CI but breaks interactive devenv), `extra-sandbox-paths` (keep empty, each path expands attack surface). Provides three complete configuration formats: NixOS module (`nix.settings` in configuration.nix), standalone `/etc/nix/nix.conf` (non-NixOS Linux, macOS), and per-user `~/.config/nix/nix.conf` with explicit tables of what can vs cannot be set per-user (daemon-enforced settings like sandbox, trusted-users, require-sigs are ignored in per-user config). Includes deployment checklist. See [detailed report](nix-conf-hardening-research.md).

### Pre-Commit Security Hook Suite (deep)
Comprehensive catalog of every security-relevant pre-commit hook for devenv.sh, with exact devenv.nix configurations, performance benchmarks, and deployment classification. Of 120+ hooks in git-hooks.nix, only ripsecrets is a dedicated security hook (built-in, 0.32s on full Sentry repo -- 95x faster than trufflehog, 226x faster than detect-secrets). All other security tools require custom hook definitions but integrate cleanly since gitleaks, trufflehog, detect-secrets, semgrep, bandit, gosec, grype, vulnix, and flake-checker are all in nixpkgs. Designed a custom lock-file-audit hook for detecting suspicious changes to devenv.lock/flake.lock (owner changes, new inputs). Classified 17 hooks into three tiers: commit-time (<5s: ripsecrets, shellcheck, bandit, lock-file-audit, flake-checker), pre-push (<30s: gitleaks, semgrep), and CI-only (trufflehog for credential verification, vulnix/grype for CVE scanning). Hook bypass via --no-verify cannot be prevented client-side; defense-in-depth requires CI running `devenv test` plus branch protection. devenv auto-reinstalls hooks on shell entry, providing stronger hook persistence than manual installation. Complete hardened hook suite devenv.nix provided. See [detailed report](precommit-hooks-research.md).

### Vulnerability Scanning Integration (deep)
Deep evaluation of 6 vulnerability scanning tools for devenv.sh integration: vulnix (Nix-native CVE scanner, actively maintained, high false positive rate from coarse name-matching heuristic but supports TOML allowlists with expiry dates), sbomnix/vulnxscan (recommended primary CI scanner -- generates CycloneDX/SPDX SBOMs from Nix closures and orchestrates three scanners simultaneously: vulnix for Nix-native matching, Grype for broader CVE database coverage, and OSV for additional sources, with multi-scanner consensus scoring), Trivy (does NOT support Nix store scanning -- issue #1673 closed/stale -- and was itself compromised in a sophisticated supply chain attack in March 2026 where attackers distributed credential-stealing malware via poisoned releases and hijacked GitHub Actions tags), Grype (consumes sbomnix CycloneDX SBOMs but misses Nix-specific packages due to the Nix PURL specification gap), nix-security-tracker (Django web app with no public API, not directly integrable, value flows indirectly through nixpkgs maintainer patches), and flake-checker (fast <5s checks for nixpkgs recency, branch support, and upstream provenance -- the only tool suitable for commit-time hooks). A fundamental gap exists: no tool can detect vulnerabilities in language-level packages (Python, Node, Rust) within Nix closures; the only workaround is building container images and scanning those with Syft+Grype. The ghafscan project provides the reference architecture for production daily Nix vulnerability scanning. Designed a 4-tier scanning strategy: commit-time (<5s: ripsecrets + flake-checker), pre-push (<30s: quick vulnix scan), CI (5-15min: vulnxscan + SBOM generation + flake-checker), nightly (15-40min: deep scan with buildtime deps, triage, three-version comparison). Includes concrete devenv.nix integration examples, GitHub Actions and GitLab CI pipeline configurations, and a comprehensive tool comparison matrix. See [detailed report](vuln-scanning-research.md).

### Runtime Isolation Options (deep)
Devenv.sh shell sessions have zero runtime isolation -- the Nix build sandbox only applies during `nix-build`, not during `devenv shell` or `devenv up`. Evaluated 7 approaches to adding runtime isolation. Two native devenv PRs exist but neither is close to merging: PR #2427 (bubblewrap, whole-shell sandbox, draft since Jan 2026, breaks shell customization, unresolved design debate) and PR #1783 (Landlock per-executable sandbox, draft since Mar 2025, author abandoned, better UX but direnv compatibility blocker from Landlock's monotonic restriction property). Manual bubblewrap wrapping works today but breaks direnv integration, SSH/GPG, and shell configs. Devcontainer generation (`devcontainer.enable = true`) is the only production-ready path to isolation, but the generated config has zero security hardening -- it needs capDrop, seccomp profiles, and read-only root filesystem added manually. A critical finding is that systemd --user sandboxing is nearly useless: PrivateTmp, ProtectSystem, and ProtectHome do NOT work with user services due to kernel namespace restrictions; only NoNewPrivileges, SystemCallFilter, and MemoryDenyWriteExecute are available. Firejail requires NixOS system-level configuration (SUID binary) and is not portable. The most promising lightweight option discovered is `landrun`, a CLI wrapper around Landlock LSM that can sandbox devenv processes today without namespace overhead, requires Linux 5.13+ (NixOS has 7.0.2), and degrades gracefully on unsupported kernels. Recommended boilerplate strategy: include hardened devcontainer config + manual bwrap wrapper script + landrun wrapper script + NoNewPrivileges for processes TODAY; track native PRs for future integration. See [detailed report](runtime-isolation-research.md).

## Open Questions (Resolved)

- **`trusted-user` status** → RESOLVED: Remove all non-root users from `trusted-users`. Use `trusted-substituters` + `trusted-public-keys` at the system level to grant scoped cache access. See [nix.conf hardening guide](nix-conf-hardening-research.md).
- **Channel pinning strategy** → RESOLVED: Pin to upstream `nixos-25.11` (stable) instead of `devenv-nixpkgs/rolling`. Accept ~2-week security patch lag vs. rolling for the benefit of broader community review. See [boilerplate](boilerplate-research.md).
- **Shell hook sandboxing** → RESOLVED (partial): Cannot be sandboxed today without breaking workflows. Mitigated by code review checklist + enterShell restricted to non-privileged operations + the trust model document for team awareness. See [trust model](trust-model-research.md). Native sandbox PRs (#2427, #1783) would address this when merged.
- **Trustix multi-party verification** → RESOLVED: Not included in boilerplate — overkill for most teams, requires infrastructure investment. Documented as a systemic improvement for high-security environments. See [prior art](prior-art-research.md).

## Open Questions (Remaining)

- Will the devenv team settle on bubblewrap or Landlock for native sandbox support? Both PRs are stalled as of May 2026. A hybrid approach would cover both whole-shell and per-executable isolation.
- Can Landlock's monotonic restriction property be worked around for direnv integration? The "peninsula" PoC using seccomp-unotify suggests a path but adds significant complexity.
- No tool can detect vulnerabilities in language-level packages (Python, Node, Rust) within Nix closures without first building a container image. Will sbomnix or the Nix PURL specification close this gap?
- How will devenv's relationship with the `devenv-nixpkgs/rolling` fork evolve? If upstream nixpkgs adoption grows, the vendor-controlled fork becomes the primary trust concern.

## Conclusions

### Answer to the Research Question

**How can devenv.sh be configured as a security-hardened boilerplate for developers?**

Devenv.sh can be meaningfully hardened through a layered defense-in-depth strategy, but with important architectural limitations that cannot be solved by configuration alone. The hardened boilerplate consists of four project-level files (devenv.yaml, devenv.nix, devenv.local.nix.example, .envrc) plus system-level nix.conf changes. No such boilerplate existed before this spike — this is novel work.

### The Security Model in One Paragraph

Nix evaluation is unsandboxed — all `.nix` code runs with the calling user's full privileges. This is the single most important security fact. The Nix build sandbox (Linux namespaces) protects during `nix-build` but does NOT apply to `devenv shell` sessions, `enterShell` hooks, or `devenv up` processes. Binary caches use Ed25519 signature verification but trust is all-or-nothing per cache. Devenv's default configuration adds a vendor-controlled nixpkgs fork and auto-trusts Cachix caches. The trust model converges on code review as the primary — and often only — security boundary.

### What the Boilerplate Achieves (6 Layers)

1. **Input hardening** (MUST-HAVE): Pin to upstream NixOS stable instead of `devenv-nixpkgs/rolling`, eliminating the vendor-controlled fork. Lock file integrity via flake.lock SHA-256 pinning.
2. **Environment isolation** (MUST-HAVE): Clean shell (`clean.enabled`) strips ambient credentials (AWS_*, GITHUB_TOKEN, etc.) from the dev environment. 35+ dangerous env vars explicitly unset.
3. **Secrets management** (MUST-HAVE): SecretSpec integration keeps secrets out of the Nix store and shell environment. Dotenv disabled to prevent accidental secret commits.
4. **Automated scanning** (RECOMMENDED): Pre-commit hooks (ripsecrets for secret detection, custom lock-file-audit for input integrity, flake-checker for nixpkgs recency). CI pipeline with vulnxscan/sbomnix for CVE scanning and SBOM generation.
5. **System hardening** (RECOMMENDED): nix.conf changes — `trusted-users = root` only, `sandbox-fallback = false`, `require-sigs = true`, explicit `trusted-substituters` allowlist. These require system admin action; devenv.nix cannot set them.
6. **Runtime isolation** (OPTIONAL/FUTURE): Devcontainer generation for teams needing container-level isolation today. Manual bubblewrap/landrun wrapper scripts for lightweight sandboxing. Native sandbox support pending (PRs #2427, #1783 stalled).

### What the Boilerplate Does NOT Protect Against

1. **Malicious code in reviewed PRs** — if a team member or contributor introduces malicious `.nix` code and it passes review, nothing stops it. Nix evaluation is unsandboxed.
2. **Upstream nixpkgs compromise** — a compromised nixpkgs commit affects all users. The stable channel provides ~2-week quarantine but is not a guarantee.
3. **Binary cache key compromise** — if `cache.nixos.org`'s signing key is compromised, require-sigs won't help. Trustix-style multi-party verification is the only defense, but is not production-ready for most teams.
4. **Language-level vulnerabilities inside Nix closures** — no tool can scan Python/Node/Rust packages within Nix derivations without building a container image first.
5. **Runtime exploitation** — no runtime sandboxing exists for dev shell sessions today. Malicious code in `enterShell` or processes has full user access.
6. **`devenv.local.nix` overrides** — local override files can silently weaken all security controls. Documented in the boilerplate but cannot be technically prevented.

### Deployment Strategy

| Tier | What | Effort | Impact |
|------|------|--------|--------|
| **Immediate** | Drop boilerplate files into projects, enable pre-commit hooks | 30 min/project | Eliminates credential leakage, secret commits, stale inputs |
| **Same week** | System-level nix.conf hardening (remove trusted-users, enforce sandbox) | 1-2 hrs/machine | Closes root-equivalent escalation path, prevents unsigned package installation |
| **First sprint** | CI pipeline with vulnxscan + SBOM generation + flake-checker | 2-4 hrs | Continuous CVE monitoring, supply chain visibility |
| **When ready** | Trust model document in team docs, PR review checklist adoption | 1-2 hrs | Human layer — developers understand what they're trusting |
| **Future** | Enable native devenv sandbox when PRs merge; adopt CA derivations when stable | Track upstream | Closes the runtime isolation gap |
