# Stacklok: From Unknown to Verified — Solving the MCP Server Trust Problem

- **Source**: https://dev.to/stacklok/from-unknown-to-verified-solving-the-mcp-server-trust-problem-5967
- **Retrieved**: 2026-05-14

## Server Trust Establishment

Trust is established through cryptographic verification of container provenance. The system verifies that "containers were built from the official GitHub repository" and "used the expected GitHub Actions workflow," using Sigstore signatures and GitHub Attestations to confirm authenticity.

## Response/Content Trust Extension

No. The article focuses exclusively on server deployment security, not response verification. There is no discussion of trusting the outputs or responses generated by MCP servers once they're running.

## Tool Response Verification Mechanisms

None mentioned. The document addresses container image integrity but makes no reference to mechanisms for verifying the integrity or authenticity of data returned by tools within those servers.

## Trust Model Structure

The model is binary per server, with configurable enforcement levels:
- Disabled (development only)
- Warning mode (verification attempted but non-blocking)
- Strict mode (verification required for deployment)

However, this remains server-level, not granular per individual response or tool invocation.

## Content Provenance/Response Integrity

Not addressed. The article explicitly states: "Provenance verification... is a practical solution to real security problems" but limits this to verifying "MCP servers actually come from their claimed sources." No mechanisms exist for tracking or verifying the provenance of responses generated during execution.
