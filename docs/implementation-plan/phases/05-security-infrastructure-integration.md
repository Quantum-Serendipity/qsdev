# Phase 5: Security & Infrastructure Integration

## Goal

Implement the cross-cutting security and infrastructure layer that connects the generated environment to the organization's security stack. This phase configures registry proxies, Nix binary caches, build caches, CI vulnerability scanning, SBOM generation, and the universal pre-commit hook suite. It also generates the nix.conf hardening guide and security documentation. Per-ecosystem package manager configs (`.npmrc`, `pip.conf`, etc.) are already generated by ecosystem modules in Phase 2 — this phase adds the **infrastructure-level** integration that connects those configs to registry proxies, caches, and scanning services.

## Dependencies

Phases 3 and 4 complete (devenv and claudecode addons can generate files).

## Phase Outputs

- Registry proxy configuration (Nexus, Artifactory, GitHub Packages, etc.) — environment variables and per-ecosystem config file sections pointing to the proxy
- Nix binary cache configuration (Cachix, Attic) — substituters and trusted-public-keys in devenv.nix/nix.conf
- Build cache configuration (sccache, ccache, Turborepo, Nx, Bazel Remote) — environment variables and tool-specific configs
- Universal pre-commit/prek hook suite (ripsecrets, secrets detection, lockfile audit, Nix linting) — cross-ecosystem baseline
- CI vulnerability scanning workflows (OSV Scanner, Socket.dev, Harden-Runner, Claude Code Security Review GitHub Action)
- nix.conf hardening guide/template
- SBOM generation configs (Syft, sbomnix)
- Dependency update configs (Renovate or Dependabot with age-gating)
- Security documentation generation (trust model, defense layers, trivy compromise warning)

### Unit 4.1: Per-Ecosystem Package Manager Security Configs

**Description:** Generate per-ecosystem security configuration files based on detected languages: .npmrc, pip.conf, pnpm-workspace.yaml, .yarnrc.yml, bunfig.toml.

**Context:** Each ecosystem has its own config file for age-gating, install script blocking, and lock file enforcement. These are the highest-impact, lowest-cost defenses per the supply chain security roadmap. All config values validated against current tool versions.

**Desired Outcome:** Language detection triggers generation of the appropriate package manager security configs with hardened defaults.

**Steps:**
1. Implement per-ecosystem config generators:
   - **npm**: `.npmrc` with `save-exact=true`, `ignore-scripts=true`, `min-release-age=3`
   - **pnpm**: `pnpm-workspace.yaml` additions with `strictDepBuilds: true`, `minimumReleaseAge: 4320`, `trustPolicy: no-downgrade`, `blockExoticSubdeps: true`
   - **yarn**: `.yarnrc.yml` with `enableImmutableInstalls: true`, `enableHardenedMode: true`, `enableScripts: false`, `npmMinimalAgeGate: 7d`
   - **bun**: `bunfig.toml` with `minimumReleaseAge = "7d"`
   - **pip/uv**: Generate CI commands reference (`pip install --require-hashes --only-binary :all:`, `uv --exclude-newer=7d`)
   - **cargo**: Generate CI commands reference (`cargo build --locked`)
   - **go**: `GOFLAGS=-mod=readonly` in devenv.nix env vars, `go mod verify` in CI
2. Wire into devenv addon: when a language is selected, include its package manager security config in generated files.
3. Add version check comments in generated configs noting minimum tool versions.
4. Add inline comments explaining each setting's security purpose.

**Acceptance Criteria:**
- [ ] npm: `.npmrc` generated with age-gating, script blocking, exact saves
- [ ] pnpm: security settings in `pnpm-workspace.yaml` format (minutes, not milliseconds)
- [ ] yarn: `.yarnrc.yml` with hardened mode and immutable installs
- [ ] bun: `bunfig.toml` with age-gating
- [ ] Go: `GOFLAGS=-mod=readonly` in devenv.nix env vars
- [ ] Each config includes version requirement comments
- [ ] Each setting has inline comment explaining security purpose

**Research Citations:**
- `research-spikes/package-supply-chain-security/quarantine-gates-research.md` — age-gating configs per ecosystem
- `research-spikes/package-supply-chain-security/install-sandboxing-research.md` — install script blocking configs
- `research-spikes/package-supply-chain-security/lockfile-integrity-research.md` — lock file enforcement
- Validation: pnpm unit confusion documented, npm exemption limitation noted

### Unit 4.2: Pre-Commit/Prek Hook Suite

**Description:** Integrate the tiered pre-commit hook suite from devenv-security into the devenv.nix generation, using prek (devenv 1.11+ default runner).

**Context:** Three tiers of hooks: baseline (always-on: ripsecrets, check-added-large-files, no-commit-to-branch), enhanced (language-aware: gofmt, prettier, shellcheck, statix), specialized (secrets scanning, lockfile audit, Nix input audit). Validation confirmed prek replaces pre-commit as default runner.

**Desired Outcome:** devenv.nix includes appropriate pre-commit hooks based on selected languages and security tier.

**Steps:**
1. Define hook tiers as configuration:
   - **Baseline** (always enabled): `ripsecrets`, `check-added-large-files`, `no-commit-to-branch`, `check-merge-conflict`
   - **Enhanced** (per-language): `gofmt` (Go), `prettier` (TS), `black`/`ruff` (Python), `rustfmt` (Rust), `shellcheck`, `statix` (Nix)
   - **Specialized** (opt-in): custom `lock-file-audit`, custom `nix-secrets-check`
2. Integrate into devenv.nix template: add `git-hooks.hooks.*` section with tier-based hooks.
3. Ensure devenv.yaml includes `git-hooks` input when any hooks are enabled (from Unit 2.1).
4. Add custom hook definitions: `lock-file-audit` (checks lockfile freshness), `nix-secrets-check` (scans for hardcoded secrets in .nix files).
5. Document prek as the runner (not pre-commit) in generated comments.

**Acceptance Criteria:**
- [ ] Baseline hooks always present when hooks enabled
- [ ] Enhanced hooks match detected languages
- [ ] Custom hooks define `entry`, `language = "system"`, `pass_filenames` correctly
- [ ] `ripsecrets` always enabled (primary secrets defense)
- [ ] `no-commit-to-branch` protects main/master by default
- [ ] Generated devenv.nix hook syntax is valid

**Research Citations:**
- `research-spikes/devenv-security/precommit-hooks-research.md` — 17 hooks in 3 tiers
- Validation: prek replaces pre-commit, git-hooks input must be explicit

**Status:** Not Started

### Unit 4.3: nix.conf Hardening Guide

**Description:** Generate a nix.conf hardening recommendation document and optionally a NixOS module snippet for system-level Nix security settings.

**Context:** nix.conf settings are system-level (not per-project), so the addon can't directly modify them. Instead, generate a recommendation document and a NixOS module snippet the admin can apply. All 10 settings validated as current.

**Desired Outcome:** A generated `docs/nix-conf-hardening.md` with copy-pasteable configurations and a NixOS module snippet.

**Steps:**
1. Generate `docs/nix-conf-hardening.md` in the project with:
   - Recommended nix.conf settings: `sandbox = true`, `sandbox-fallback = false`, `require-sigs = true`, `trusted-users = root`, `filter-syscalls = true`, `accept-flake-config = false`
   - NixOS module snippet (for NixOS users)
   - Standalone nix.conf snippet (for non-NixOS)
   - Per-user nix.conf guidance (what can/cannot be set)
2. Include trust model summary: which caches are trusted, which keys verify them.
3. Include warning about `trusted-users` (root-equivalent).
4. Make generation optional (gated by `--nix-hardening-guide` flag or wizard question).

**Acceptance Criteria:**
- [ ] Generated document includes all 10 nix.conf settings
- [ ] NixOS module snippet is valid Nix syntax
- [ ] `trusted-users` warning is prominent
- [ ] Document is standalone-readable for an admin who hasn't seen the research
- [ ] Generation is opt-in (not every project needs this)

**Research Citations:**
- `research-spikes/devenv-security/nix-conf-hardening-research.md` — all 10 settings with rationale
- `research-spikes/devenv-security/trust-model-research.md` — trust chain analysis
- Validation: all settings confirmed valid against Nix 2.33/2.34

**Desired Outcome:** Ready-to-use CI workflow snippets for the recommended scanning stack.
