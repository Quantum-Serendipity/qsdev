## 2026-05-14 17:00 — DevDocs Architecture Deep Dive Complete
- **Type**: research
- **Status**: success
- **Depth**: deep
- **Sources**: 19 web sources saved to `docs/` (see `devdocs-research.md` for full list)
- **Summary**: Comprehensive research into DevDocs internals, self-hosting, data format, MCP servers, alternatives, and security properties. Key findings: (1) DevDocs data format is trivially simple — three JSON files per doc set (`index.json` for search, `db.json` for content, `meta.json` for metadata). No database needed, just filesystem access. (2) Docker image with ALL docs is ~3.5 GB compressed (docs layer is 99%+ of size); selective download dramatically reduces this. (3) DevDocs has NO REST API — all existing MCP servers either access static JSON files directly or wrap the web UI. (4) Six DevDocs MCP servers exist; `jiegec/devdocs-mcp-server` (Python, extracts from Docker image, no running server) and `madhan-g-p/DevDocs-MCP` (NestJS, version-pinning from package.json) are strongest. (5) Context7 is unsuitable for gdev — cloud-only, proprietary backend, demonstrated ContextCrush vulnerability. (6) Zeal/Dash use a different format (SQLite) with no MCP servers. (7) Prompt injection risk is genuinely reduced by local DevDocs: content is scraped once from official sources, sanitized (scripts/styles stripped), versioned, and auditable. Residual risk is upstream compromise at scrape time. (8) Recommended architecture: direct file access to DevDocs JSON files from a TypeScript MCP server, no DevDocs web server needed.

- **Status**: success

- **Summary**: Deep investigation across 6 areas: (1) MCP server chaining/routing — Claude Code provides NO native priority routing between MCP servers; tool search presents a flat catalog; the MCP protocol has no routing/fallback primitives; the entire MCP proxy ecosystem (MetaMCP 2.3k stars, FastMCP, Envoy AI Gateway, combine-mcp, 6+ others) implements aggregation only, not priority-based failover. (2) Skill-level vs MCP-level routing — comprehensive tradeoff analysis of both approaches across 12 factors; skill-level wins on build effort (~1 hour vs ~1-2 weeks), failure modes (graceful vs catastrophic), latency (zero vs +300-400ms), prompt injection surface (minimal vs moderate), and adaptability (Claude reasons contextually vs fixed routing logic); MCP-level wins only on determinism and testability. (3) "No results" detection — DevDocs: empty index search is unambiguous miss; ZIM/Xapian: BM25 scores are relative not absolute, `get_percent()` < 15-20% heuristic for miss; man pages: binary hit/miss; adapted CRAG three-state model (CORRECT/AMBIGUOUS/INCORRECT) where Claude IS the relevance evaluator in skill-level routing. (4) Graceful degradation — five scenarios analyzed (stale docs, missing library, air-gapped, conflicting sources, MCP crash) with skill-level handling for each. (5) Prior art — CRAG (arXiv 2401.15884) three-state classification with +0.431 context_precision improvement; metasearch cascade pattern; DNS-style fallback chains; all map to skill-level priority ordering. (6) Concrete architecture — full component diagram, .mcp.json configuration, complete SKILL.md definition with dynamic context injection via `!gdev docs status`, source tagging rules, degradation instructions, and 4-phase evolution path from skill to meta-MCP.
- **Recommendation**: Skill-level routing as primary architecture. Zero infrastructure, leverages Claude's contextual reasoning, stronger security model (separate failure domains), incrementally improvable. Meta-MCP as future phase if skill proves unreliable.

- **Summary**: Comprehensive research into Azure self-hosting for large documentation corpora (~100 GB ZIM + DevDocs) with Entra ID SSO. Investigated 9 areas: (1) Azure storage options — compared Blob Storage, Azure Files, and NetApp Files for random access patterns; Blob Storage + BlobFuse2 is optimal (cheapest at ~$1.80/month for 100 GB, BlobFuse2 file cache provides local-SSD performance after first download). (3) Entra ID authentication — DefaultAzureCredential chain picks up `az login` tokens automatically; credential chain: EnvironmentCredential → WorkloadIdentity → ManagedIdentity → VS Code → Azure CLI → PowerShell → Developer CLI. (4) Terraform module design — complete module with storage account, containers, RBAC (Storage Blob Data Reader), private endpoints, and gdev profile integration; shared_access_key_enabled=false forces Entra-only auth. (5) kiwix-serve as Azure Container Apps — built-in Entra ID auth, scale-to-zero possible, ~$36-72/month always-on; adds 20-50ms latency vs local. (6) DevDocs on Azure — JSON files in Blob Storage accessed via BlobFuse2 or Azure SDK; no need to run DevDocs web server. (7) Developer experience — az login once, BlobFuse2 auto-mounts via systemd, tokens auto-refresh for ~90 days, offline fallback to local cache. (8) Cost analysis — $5-8/month for BlobFuse2 approach (20 developers), $46-51/month for always-on kiwix-serve, $15-25/month for scale-to-zero. (9) Network security — public + IP restriction sufficient for public documentation content; private endpoints available for enterprise requirements.
- **Key finding**: BlobFuse2 is the linchpin — it makes Azure Blob Storage look like a local filesystem to openzim-mcp and devdocs-mcp with zero code changes, while providing Entra ID authentication and local SSD performance after caching. Already packaged in nixpkgs.

## 2026-05-14 23:30 — Multi-Cloud Terraform Abstraction Research Complete
- **Type**: research
- **Status**: success
- **Depth**: deep
- **Sources**: 13 web sources saved to `docs/` (see `multi-cloud-terraform-research.md` Sources section for full list)
- **Summary**: Comprehensive research into making documentation hosting cloud-agnostic across AWS, GCP, Azure, and beyond. Investigated 9 areas: (1) AWS architecture — S3 Standard ($2.30/mo for 100 GB) with Mountpoint for S3 (6-8x faster than s3fs) or rclone mount; AWS IAM Identity Center SSO with auto-refreshing token cache; ECS Fargate for hosted kiwix-serve ($18-65/mo). (2) GCP architecture — Cloud Storage Standard ($2.00/mo) with gcsfuse or rclone; Filestore is overkill (1 TB minimum, $200+/mo); ADC auto-detects gcloud auth login; Cloud Run for hosted service ($35-45/mo); IAP provides free SSO gateway. (3) Other providers — DigitalOcean Spaces ($5/mo flat for 250 GB), Hetzner Object Storage (~$0.52/mo for 100 GB, cheapest), Oracle Cloud (20 GB free forever + free ARM VMs), MinIO self-hosted (AGPL license concern). (4) Terraform abstraction patterns — per-provider modules with common interface contract is the standard (CloudPosse, Gruntwork, terraform-aws-modules all AWS-only); HashiCorp explicitly advises against cloud-agnostic modules; CDKTF interesting for gdev's TypeScript stack but adds complexity. (5) Storage abstraction — S3 API covers ~90% of providers (AWS, GCS via HMAC, DO, Hetzner, Oracle, MinIO, etc.); Azure is the gap requiring BlobFuse2. Two-path strategy: S3-compatible (default) + Azure-native. (6) SSO/IAM — all clouds use identical credential chain cascade (env vars → CLI token → managed identity); gdev should detect configured cloud CLI, not abstract auth. Vault is overkill. (7) Hosted service — single Docker image works across Fargate/Cloud Run/ACI; FUSE-mount approach (zero compute cost) should be default; hosted service is optional add-on. (8) Module design — concrete Terraform structure with 5 provider directories (aws/, azure/, gcp/, s3-compatible/, local/) sharing common variable/output contracts; gdev generates the right one from profile selection. (9) Air-gapped — MinIO on-prem, NFS share, or direct disk; same MCP config works with any mount path.
- **Key findings**: (a) gdev itself is the abstraction layer, not Terraform — generate per-provider modules from templates. (b) rclone mount with --vfs-cache-mode full is the universal FUSE tool (50+ backends, sparse file caching for ZIM random I/O). (c) S3 API covers most providers; Azure is the exception requiring dedicated support. (d) Storage costs are trivial ($0.52-5/mo); hosted service is where costs vary ($25-65/mo). (e) The MCP server config is identical regardless of provider — only the mount command changes.
