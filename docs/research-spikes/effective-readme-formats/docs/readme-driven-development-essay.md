# Readme Driven Development
- **Source**: https://tom.preston-werner.com/2010/08/23/readme-driven-development.html
- **Retrieved**: 2026-05-15
- **Author**: Tom Preston-Werner (GitHub co-founder)

## Core Argument

Tom Preston-Werner's central thesis is that "A perfect implementation of the wrong specification is worthless," and similarly, undocumented software lacks value. He advocates writing project documentation as the primary creative act before any coding begins.

## Key Problems Identified

Preston-Werner identifies several issues with current development practices:

- Software may solve the wrong problem or be incomprehensible to users
- The pendulum swung too far from waterfall's excessive documentation to agile's minimal specs
- Projects lack adequate or entirely missing introductory documentation
- Retroactive documentation writing is burdensome and incomplete

## Recommended Practices

**Write the README first.** This means completing it before writing "any code or tests or behaviors or stories or ANYTHING."

The practice yields specific advantages:

1. **Thoughtful design**: You clarify ideas without constant code refactoring overhead, similar to discovering test-driven development's benefits

2. **Built-in documentation**: Documentation quality improves when written during peak project enthusiasm, avoiding the "absolute drag" of retroactive writing

3. **Team coordination**: Documented interfaces enable parallel development rather than serial coding chains

4. **Concrete discussion**: Written specifications replace circular conversations with something tangible to critique and refine

Preston-Werner positions README-driven development as occupying the "middle ground" between exhaustive specifications and no specifications — maintaining agility while ensuring clarity and intentional design.
