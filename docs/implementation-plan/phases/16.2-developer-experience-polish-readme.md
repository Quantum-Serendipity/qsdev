# Phase 16.2: Developer Experience Polish — README

## Goal

Create a README.md for the gdev GitHub repository that functions as a conversion funnel, applying the 12 winning patterns and avoiding all 11 anti-patterns identified in the effective-readme-formats research spike. The README is the single most important document for driving gdev adoption — it must move a skeptical developer from first scan (0-5s) through quick evaluation (5-30s) to trying the tool (30s-5min) to first value (5-15min). This phase is documentation only — no code changes to gdev itself.

## Dependencies

Phase 10 complete (distribution and self-bootstrapping — need to know all installation methods, install script URLs, package manager availability, and `nix run` invocation syntax). Phase 16 complete (developer experience polish — need the final command set including `gdev init`, `gdev doctor`, `gdev repair`, `gdev info`, `gdev status`, `gdev update`, `gdev teardown`, and `gdev outdated` for accurate quick-start and feature documentation).

## Phase Outputs

- `README.md` for the gdev repository root, structured as a conversion funnel
- Logo/wordmark asset (SVG, with dark/light mode variants)
- Terminal recording assets (asciinema recordings converted to SVG/GIF)
- Before/after comparison visual showing manual setup vs `gdev init`

### Unit 16.2.1: README Content Architecture

**Description:** Design the complete README structure by mapping the conversion funnel model to specific sections, defining the above-the-fold content strategy, establishing the progressive disclosure hierarchy, and assigning each of the 12 winning patterns to its target section.

**Context:** The research establishes that a README is a landing page, not a manual. The reader has not committed to anything and will decide within ~60 seconds whether to invest further. The F-pattern scanning research (NNGroup, replicated 2006-2024) shows the first line after the title receives the most attention, and 80% of viewing time is spent above the fold. The recommended structure must serve all three developer learning styles simultaneously: systematic (top-to-bottom reading), opportunistic (jump to quick-start and copy-paste), and pragmatic (scan headings, deep-dive selectively).

**Desired Outcome:** A documented content architecture that specifies exact section order, word budget per section, which patterns map to which sections, and the progressive disclosure strategy (Level 1 visible, Level 2 in `<details>` blocks, Level 3 linked to external docs). The architecture document serves as the blueprint for all subsequent units.

**Steps:**
1. Define the four-layer funnel mapping for gdev's README:
   - **Above the Fold (0-5s scan):** Logo/wordmark, one-liner value proposition, strategic badges (3-5), hero visual (terminal recording of `gdev init`). Target: first screenful, ~25-30 rendered lines.
   - **The Quick Win (5-30s evaluation):** Before/after comparison (manual setup vs `gdev init`), benefit-framed feature bullets (5-7 outcomes), 30-second quick-start (3-5 commands). Target: second screenful.
   - **The Depth Layer (30s-5min for committed readers):** Platform-aware installation (primary visible, others in `<details>`), try-before-install (`nix run`), ecosystem integration table, detailed usage examples (basic, with options, real-world).
   - **Trust & Community (bottom):** Honest positioning / non-goals section, documentation links, contributing link, license.
2. Map each of the 12 winning patterns to its target section:
   | Pattern | Target Section | Implementation |
   |---------|---------------|----------------|
   | P1: One-liner value proposition | Above the fold | First line after title: under 15 words, names what gdev replaces |
   | P2: Visual proof before text | Above the fold | Terminal recording of `gdev init` immediately after badges |
   | P3: Concrete comparison | Quick Win | Side-by-side before/after: 7 manual steps vs 1 command |
   | P4: 30-second quick-start | Quick Win | `gdev init` + `direnv allow` + show generated files |
   | P5: Platform-aware installation | Depth Layer | Primary method visible, 3+ in `<details>` blocks |
   | P6: Benefit-framed bullets | Quick Win | 5-7 outcomes, bold keyword + one-line explanation |
   | P7: Calibrated social proof | Above the fold | Badges only (build, version, license, platforms); testimonials later if available |
   | P8: Emotional hook | Quick Win | Problem framing: "Manual devenv.nix, scattered Claude Code settings, missing security configs..." |
   | P9: Ecosystem positioning | Depth Layer | Integration table: devenv.sh, Claude Code, pre-commit, 27 language ecosystems |
   | P10: Configuration-as-quickstart | Quick Win | Show the generated `devenv.nix` and `CLAUDE.md` — gdev's output IS the config |
   | P11: Try-before-install | Depth Layer | `nix run github:Quantum-Serendipity/gdev -- init` |
   | P12: Honest positioning | Trust & Community | Explicit non-goals list citing the 13 confirmed non-expansions |
3. Define the progressive disclosure hierarchy:
   - **Level 1 (visible in README):** One-liner, hero visual, comparison, quick-start, primary install method, feature bullets, non-goals.
   - **Level 2 (one click via `<details>`):** Alternative installation methods per platform, full ecosystem support matrix, advanced configuration examples, generated file samples.
   - **Level 3 (linked out):** Full documentation site, architecture docs, CONTRIBUTING.md, CHANGELOG.md, individual command reference.
4. Set word budget constraints: above-the-fold content must render in under 30 lines. The entire README (excluding collapsed `<details>` content) must stay under 500 words of prose. Total rendered length including code blocks: under 150 lines before the first `<details>` section.
5. Verify the architecture against all 11 anti-patterns to confirm none are present:
   - A1 (empty README): Comprehensive content planned.
   - A2 (sponsors above fold): No sponsor content above the value proposition.
   - A3 (installation deferred): Primary install method in the README itself.
   - A4 (missing quick-start): Dedicated quick-start with copy-paste commands.
   - A5 (README as docs index): Link to docs site, not a sitemap.
   - A6 (wall of text): Progressive disclosure via `<details>`, visual breaks, code blocks.
   - A7 (feature-first framing): Problem-first framing in the emotional hook and comparison.
   - A8 (stale examples): Examples generated from actual gdev output; CI validation planned.
   - A9 (inflated benchmarks): Time savings stated as ranges, not precise claims, with methodology noted.
   - A10 (assumed context): No jargon without explanation; README works for first-time viewers.
   - A11 (missing license): License section at bottom with SPDX identifier.
6. Document the architecture as a section-by-section outline in a working document that subsequent units reference during content creation.

**Acceptance Criteria:**
- [ ] Section order documented with exact sequence matching the conversion funnel model
- [ ] Each of the 12 winning patterns assigned to a specific section with implementation notes
- [ ] Progressive disclosure hierarchy defined with content assigned to Level 1 / Level 2 / Level 3
- [ ] Word budget set: under 500 words of prose visible without expanding any `<details>` block
- [ ] All 11 anti-patterns verified absent from the architecture
- [ ] Architecture document exists and is referenced by subsequent units

### Unit 16.2.3: Concrete Comparison & Value Demonstration

**Description:** Create the side-by-side "Before gdev" vs "After gdev" comparison, benefit-framed feature bullets, and problem-framing emotional hook that together form the Quick Win layer of the conversion funnel. This is the content that answers "Is this better than what I have?" within 5-30 seconds.

**Context:** The concrete comparison is the single most effective persuasion element identified across all 20 tools analyzed in the research. fd's syntax comparison (`fd PATTERN` vs `find -iname '*PATTERN*'`) and uv's "replaces pip, pip-tools, pipx, poetry, pyenv, twine, virtualenv" are the gold standards. Yet only 6/20 tools include a concrete comparison -- making this the biggest missed opportunity identified in the study. For gdev, the comparison is especially powerful because the manual alternative involves 7+ separate configuration files across 3+ tools, making the contrast stark.

**Desired Outcome:** A reader who has passed the 5-second scan now spends 5-30 seconds evaluating. After this section, they understand: (1) what they currently do manually that gdev automates, (2) the specific outcomes gdev delivers (not capabilities), and (3) whether their situation matches gdev's target use case.

**Steps:**
1. Create the "Before gdev" vs "After gdev" side-by-side comparison:
   ```markdown
   ## Before gdev vs After gdev

   <table>
   <tr><th>Without gdev (manual setup)</th><th>With gdev</th></tr>
   <tr>
   <td>

   1. Write `devenv.nix` from scratch (50-200 lines)
   2. Create `.envrc` with `use devenv`
   3. Configure `devenv.yaml` with inputs
   4. Set up `.pre-commit-config.yaml` hooks
   5. Write Claude Code `settings.json` deny rules
   6. Create `CLAUDE.md` with project conventions
   7. Add `.gitignore` entries for `.devenv/`, etc.
   8. Configure MCP servers in `.mcp.json`
   9. Set up security configs per ecosystem

   **Time: 30-90 minutes per project**

   </td>
   <td>

   ```bash
   gdev init
   ```

   **Time: under 2 minutes**

   </td>
   </tr>
   </table>
   ```
   - The left column must list real, specific manual steps that developers recognize. These are drawn directly from gdev's generation targets across Phases 2-6.
   - The right column shows the single command. The contrast between a numbered list and a single code block is visually immediate.
   - Include time estimates. These should be honest ranges, not precise claims (per the benchmark credibility research). "30-90 minutes" for manual setup reflects variation by project complexity. "Under 2 minutes" for gdev init includes wizard interaction time.
2. Write the emotional hook / problem framing paragraph:
   - Place immediately before or after the comparison.
   - Use the problem-solution copywriting framework: name the frustration, then present gdev as the resolution.
   - Draft: "Every new project starts the same way: write a devenv.nix, copy Claude Code settings from the last project, remember which pre-commit hooks you need, set up security configs you'll forget about. gdev does all of it in one command."
   - The tone must be authentic and developer-relatable (per lazygit's Elevator Pitch pattern), not corporate-marketing. Name specific artifacts developers recognize.
3. Write 5-7 benefit-framed feature bullets:
   - Use the bold-keyword + one-line-explanation format (per fzf's four-bullet pattern and starship's six-bullet pattern).
   - Frame as outcomes the developer experiences, not technical capabilities gdev has.
   - Draft bullets:
     - **Instant setup** -- Generate a complete devenv.nix, security configs, and Claude Code settings in under 2 minutes
     - **27 ecosystems** -- TypeScript, Python, Go, Rust, Java, .NET, Docker, Terraform, and 19 more, each with tailored packages and security hardening
     - **Security by default** -- Pre-commit hooks, package manager lockfile enforcement, age-gating for new dependencies, and AI agent deny rules, all generated from your project's actual stack
     - **Team-portable** -- Commit `.gdev.yaml` and teammates run `gdev init --join` to get identical environments
     - **Non-destructive** -- Detects existing configs and merges gdev sections alongside your customizations, never overwrites
     - **Self-healing** -- `gdev doctor` diagnoses environment drift, `gdev repair` fixes it automatically
     - **Claude Code-native** -- Generated CLAUDE.md, settings.json deny rules, MCP server configs, and agentic skills purpose-built for AI-assisted development
   - Verify each bullet names a user-facing outcome. Remove any that read as feature-listing rather than problem-solving.
4. Optionally include a "replaces N tools/steps" one-liner (uv/Ruff pattern):
   - "One command replaces manual devenv.nix authoring, Claude Code configuration, pre-commit setup, security hardening, and .gitignore management."
   - This answers "why would I switch?" and "what manual work can I eliminate?" simultaneously.
5. Show what the generated output looks like (configuration-as-quickstart pattern from delta):
   - Include a collapsed `<details>` block showing a representative generated `devenv.nix` (10-15 lines, not the full file) and a representative `CLAUDE.md` section (5-10 lines).
   - This serves opportunistic learners who want to see the concrete output before deciding to try it.

**Acceptance Criteria:**
- [ ] Before/after comparison uses a two-column table with specific, recognizable manual steps on the left and `gdev init` on the right
- [ ] Time estimates are honest ranges, not precise claims (e.g., "30-90 minutes" vs "under 2 minutes")
- [ ] Emotional hook paragraph uses problem-solution framing, names specific artifacts developers recognize, and avoids corporate marketing language

### Unit 16.2.4: Quick-Start & Installation Section

**Description:** Write the 30-second quick-start (3-5 copy-paste commands producing a visible result), the platform-aware installation section with progressive disclosure, and the try-before-install option. This is the content that answers "Can I get this working easily?" and makes the 15-minute TTV window feel achievable.

**Context:** The 15-minute TTV rule (daily.dev research) establishes that developers abandon tools that fail to demonstrate value within 15 minutes of initial use. The README's quick-start section functions as a promise: "You will achieve [specific outcome] in [time estimate]." fzf's three-keybinding pattern and Deno's 3-step web server are the best quick-start exemplars. For installation, zoxide's 4-step numbered flow with collapsible platform sections was ranked the best installation flow across all 20 tools. eza's `nix run` try-before-install pattern enables zero-commitment evaluation.

**Desired Outcome:** A developer reading the quick-start can copy-paste 3-5 commands and have a fully generated development environment within 2 minutes. The installation section provides exactly one primary method visibly, with alternatives in collapsible sections. The try-before-install option allows evaluation without any local installation.

**Steps:**
1. Write the 30-second quick-start section:
   ```markdown
   ## Quick Start

   ```bash
   # Install gdev (see below for other methods)
   curl -LsSf https://get.gdev.dev | sh

   # Initialize a new project
   cd your-project
   gdev init

   # Activate the environment
   direnv allow
   ```

   After `gdev init`, your project has:
   ```
   devenv.nix          # Nix development environment with your detected ecosystems
   devenv.yaml         # devenv inputs and configuration
   .envrc              # direnv integration
   .pre-commit-config.yaml  # Security and quality hooks
   .claude/settings.json    # Claude Code deny rules and permissions
   CLAUDE.md           # AI agent project context (gdev-managed sections)
   .mcp.json           # MCP server configuration
   .gitignore          # Updated with .devenv/, .gdev/ entries
   ```
   ```
   - The quick-start INCLUDES an install command as the first step (per the research finding that quick-start should precede or be interleaved with installation to create motivation).
   - Show the expected output: a file listing of what was generated. This proves the tool works and is more compelling than prose description.
   - Maximum 5 commands. Each command must be copy-paste ready with no placeholder values that require editing.
   - The `cd your-project` line is the only user-specific value. Consider whether to use a concrete example project name instead.
2. Write the try-before-install section:
   ```markdown
   ## Try Without Installing

   ```bash
   nix run github:Quantum-Serendipity/gdev -- init
   ```

   Requires [Nix](https://nixos.org/) with flakes enabled. Runs gdev directly from the repository without installing it.
   ```
   - This is the lowest-friction evaluation path. It enables a developer to try gdev in a throwaway directory without any permanent installation.
   - Include a brief prerequisite note (Nix with flakes). Do not over-explain -- the target audience for `nix run` already knows what this means.
3. Write the platform-aware installation section:
   ```markdown
   ## Install

   ```bash
   curl -LsSf https://get.gdev.dev | sh
   ```

   <details><summary>Other installation methods</summary>

   ### Homebrew (macOS / Linux)
   ```bash
   brew install quantum-serendipity/tap/gdev
   ```

   ### Nix
   ```bash
   nix profile install github:Quantum-Serendipity/gdev
   ```

   ### From source
   ```bash
   go install github.com/Quantum-Serendipity/gdev@latest
   ```

   ### Binary download
   Pre-built binaries for Linux (amd64, arm64), macOS (amd64, arm64), and Windows (amd64) are available on the [Releases page](link).

   </details>
   ```
   - Lead with the single most universal installation method (curl installer from Phase 10).
   - All alternative methods go in a single `<details>` block, not one per platform.
   - Include platform-specific gotcha warnings where relevant (e.g., WSL2 requirements, Nix flakes prerequisite).
   - The installation section is deliberately placed AFTER the quick-start, following the research finding that showing what the tool does before asking you to install it is more effective (uv, bat, fd all do this).
4. Add prerequisites note:
   - gdev generates Nix-based environments, so Nix and direnv must be installed. If `gdev doctor` (Phase 9) can detect and guide installation of these prerequisites, mention that: "Don't have Nix or direnv? Run `gdev setup` to install prerequisites automatically."
   - Keep this brief -- one line, not a multi-step guide. Link to Phase 9's `gdev setup` for details.
5. Verify the quick-start works end-to-end in a fresh environment by tracing the steps against gdev's actual command behavior from the implementation plan phases.

**Acceptance Criteria:**
- [ ] Quick-start section contains exactly 3-5 copy-paste commands
- [ ] No placeholder values requiring user editing (except `cd your-project` which is unavoidable)
- [ ] Expected output shown: list of generated files with one-line descriptions
- [ ] Try-before-install section provides a working `nix run` command
- [ ] Installation section leads with ONE primary method (curl installer)
- [ ] All alternative installation methods are in a single collapsible `<details>` block
- [ ] Quick-start appears before the detailed installation section in the README
- [ ] Prerequisites (Nix, direnv) mentioned with a path to automated setup (`gdev setup`)
- [ ] A developer following the quick-start can reach first value (generated config files) in under 2 minutes of terminal time

### Unit 16.2.5: Ecosystem & Social Proof
