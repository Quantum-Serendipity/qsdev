# Apache-2.0 License Impact on gdev Addons — Synthesis

gdev (`fastcat.org/go/gdev`) is Apache License 2.0. This document synthesizes research into how that license affects our custom addon code, binary distribution, and generated output files. Detailed backing research is in the three companion reports.

**Disclaimer**: This is a technical analysis based on the license text and published legal commentary, not legal advice. Consult a qualified attorney for binding guidance.

---

## Addon Source Code — No Constraints from gdev's License

Apache-2.0 Section 1 explicitly carves out works that "merely link (or bind by name) to the interfaces of" the Work from its "Derivative Works" definition. Our addons import gdev's public API — that's exactly what this carve-out was designed for. Go's static linking doesn't change this; the license text uses "link" without distinguishing static vs dynamic.

- Addons can be licensed under **any license** (Apache-2.0, MIT, proprietary, etc.)
- No Apache-2.0 headers needed in addon source files
- No obligation to share addon source code
- No permission needed from gdev's authors
- **Recommended**: Apache-2.0 (ecosystem alignment with K8s/Go conventions, patent protection), with MIT as a simpler alternative

### Licenses to Avoid in Go

| License | Issue |
|---|---|
| GPL-2.0 | Incompatible with Apache-2.0 (both ASF and FSF agree) |
| LGPL (any version) | Go's static linking makes compliance impossible — use MPL-2.0 instead |
| AGPL-3.0 | Technically compatible but toxic to enterprise adoption; Google bans entirely |
| GPL-3.0 | Legal but forces entire binary to GPL-3.0, limiting downstream adoption |

## Binary Distribution — Lightweight Attribution Obligations

The compiled Go binary contains gdev's code, so Section 4 redistribution obligations apply:

| Obligation | Required? | Notes |
|---|---|---|
| Include Apache-2.0 license text | **Yes** | Ship alongside binary |
| Include gdev's NOTICE file | **Only if one exists** | gdev currently has no NOTICE file |
| Mark modified gdev files | **Only if we modify gdev** | Not applicable when using as-is |
| Handle transitive deps | **Yes** | Use `google/go-licenses` to automate |

### Tooling

- `go-licenses check ./cmd/your-binary` in CI — catches incompatible transitive deps
- `go-licenses save ./cmd/your-binary --save_path=./third-party-licenses` at release — auto-generates compliance artifacts

## Generated Output Files — Varies by Type

Our addons generate files into user projects. Whether Apache-2.0 applies depends on the nature of the output:

| Generated File | Apache-2.0 Applies? | Rationale |
|---|---|---|
| `devenv.nix` / `devenv.yaml` | **No** | Functional config, transformation of user input, below originality threshold |
| `.envrc` | **No** | Trivial (1-5 lines), merger doctrine applies |
| `settings.json` | **No** | Machine-generated key-value pairs, not copyrightable |
| `CLAUDE.md` | **Partially** | Template prose carries the license; variable-substituted portions don't |
| Skills files (embed.FS copies) | **Yes** | Verbatim copies of Apache-2.0-licensed content |

### The Key Distinction

- **Code-logic-generated config** (marshaling structs to YAML/JSON from user wizard input): transformation of user input, not covered by gdev's license.
- **Verbatim-copied files** from `embed.FS` (skills, rules): literal copies of the Apache-2.0 work, license travels with them.

Since Apache-2.0 is permissive, even for files where the license does apply, users can freely use/modify/redistribute — the only obligation is attribution.

### Precedent

- **Kubebuilder** (Apache-2.0, Go scaffolding): defaults to Apache-2.0 headers on generated files but provides `--license` and `--license none` flags for user control.
- **Create-React-App** (MIT): generated projects have NO license — treated as entirely user-owned.
- **Cookiecutter** (BSD-3): injects user's name into generated copyright notices.
- **Terraform scaffolding**: ambiguous documentation caused community confusion — cautionary example.

## Patent Protection

By using gdev, we receive a patent grant from gdev's contributors covering their patents on gdev's code. We don't owe any patent grants on our own addon code. The retaliation clause only triggers if we sue claiming gdev itself infringes our patent — defensive litigation doesn't trigger it.

---

## Recommended Practices for This Project

1. **License our addons under Apache-2.0** — matches gdev, K8s ecosystem, includes patent protection
2. **Add `<!-- SPDX-License-Identifier: Apache-2.0 -->` to embed.FS skills files** — clear provenance for verbatim-copied content
3. **Add `# Generated by gdev` comments to config files** — no license header needed (functional, not copyrightable)
4. **Document generated output licensing** in README — prevents Terraform-style confusion
5. **Provide a `--license` flag** (kubebuilder pattern) — lets users control license headers on generated files
6. **Run `go-licenses check` in CI** — catches GPL-2.0 or other incompatible transitive deps early
7. **Run `go-licenses save` at release** — auto-generates third-party license bundle for binary distribution
