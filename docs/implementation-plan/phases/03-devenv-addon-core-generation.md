# Phase 3: devenv Addon — Core Generation

## Goal

Implement the devenv addon's file generation: security-hardened devenv.yaml via struct marshaling, devenv.nix via text/template **composing fragments from ecosystem modules**, and .envrc generation. The devenv addon doesn't hardcode language support — it queries registered ecosystem modules for their devenv.nix fragments, pre-commit hooks, and devenv.yaml inputs, then composes them into the final output. At the end of this phase, the devenv addon can generate a complete, security-hardened, working devenv environment for any combination of Tier 1 ecosystems.

## Dependencies

Phases 1 and 2 complete (shared types, template engine, generation pipeline, Tier 1 ecosystem modules).

## Phase Outputs

- devenv.yaml generation with hardened defaults from devenv-security spike, composing inputs from ecosystem modules
- devenv.nix main template composing language/service fragments from ecosystem modules (not hardcoded sub-templates)
- devenv.nix service sub-templates (PostgreSQL, Redis, MySQL, MongoDB, Elasticsearch, RabbitMQ) — services remain in the devenv addon since they're devenv-specific, not ecosystem-specific
- Per-ecosystem security config files generated by ecosystem modules (`.npmrc`, `pip.conf`, `settings.xml`, etc.)
- .envrc generation (direnv) and native activation guidance
- Security defaults integration (clean environment, unsetEnvVars, dotenv disabled, impure disabled)
- Four CLI commands: `qsdev devenv init`, `update`, `add-service`, `add-language`

### Unit 2.1: devenv.yaml Generation with Hardened Defaults

**Description:** Implement struct-marshaled devenv.yaml generation with security-hardened defaults from the devenv-security boilerplate.

**Context:** devenv.yaml is machine-owned (safe to regenerate). Uses Go struct marshaling to guarantee valid YAML. Hardened defaults include: `clean.enabled: true`, `impure: false`, `nixpkgs.allow_unfree: false`, `nixpkgs.allow_broken: false`. Must include explicit `git-hooks` input when hooks are enabled (critical devenv 2.0 requirement).

**Desired Outcome:** `DevenvYamlGenerator.Generate(answers)` produces a valid, security-hardened devenv.yaml.

**Steps:**
1. Define `DevenvYaml` struct with all fields (inputs, imports, nixpkgs, clean, impure, secretspec, require_version).
2. Implement `buildDevenvYaml(answers WizardAnswers) DevenvYaml` — populate from wizard answers with hardened defaults.
3. Always include `nixpkgs` input. Include `git-hooks` input when any hooks are enabled.
4. Include language-specific inputs (e.g., nixpkgs-python for Python version pinning).
5. Set hardened defaults: `clean.enabled: true`, `clean.keep` list from devenv-security boilerplate, `impure: false`, `nixpkgs.allow_unfree: false`.
6. Marshal via `yaml.Marshal()` and wrap in `GeneratedFile` with `Overwrite` strategy.
7. Write unit tests with multiple project-type inputs.

**Acceptance Criteria:**
- [ ] Generated YAML is valid (round-trip test)
- [ ] `clean.enabled: true` is always present
- [ ] `impure: false` is always present
- [ ] `git-hooks` input is included when hooks are enabled, absent otherwise
- [ ] Language-specific inputs are included for Python
- [ ] nixpkgs pin version is configurable (defaults to latest stable)

**Research Citations:**
- `research-spikes/gdev-extension-design/devenv-addon-design.md § devenv.yaml struct` — DevenvYaml struct definition
- `research-spikes/devenv-security/boilerplate-research.md § devenv.yaml` — hardened defaults

### Unit 2.2: devenv.nix Main Template & Language Sub-Templates

**Description:** Implement the devenv.nix Nix template with sub-templates for Go, TypeScript, Python, and Rust.

**Context:** devenv.nix is generated via `text/template` with the Nix FuncMap from Phase 1. The main template composes sub-templates for each selected language. Security hardening includes: `dotenv.enable = false`, `unsetEnvVars` with 30+ credential variables, explicit packages list.

**Desired Outcome:** Template renders valid, security-hardened devenv.nix for any combination of supported languages.

**Steps:**
1. Create `addons/devenv/templates/devenv.nix.tmpl` — main template with conditional sections (packages, env vars, languages, services, scripts, hooks, enterShell).
2. Create language sub-templates: `templates/languages/go.nix.tmpl`, `typescript.nix.tmpl`, `python.nix.tmpl`, `rust.nix.tmpl`.
3. Embed security defaults: `dotenv.enable = false`, `unsetEnvVars` list from devenv-security boilerplate (AWS_*, GITHUB_TOKEN, etc.).
4. Implement `DevenvNixTemplateData` struct as the bridge from `WizardAnswers` to template data.
5. Wire through template engine from Phase 1.
6. Test: render Go+PostgreSQL template, validate with `nix-instantiate --parse`.

**Acceptance Criteria:**
- [ ] Go template renders valid Nix with `languages.go.enable = true` and version
- [ ] TypeScript template handles npm/pnpm/yarn/bun package manager selection
- [ ] Python template handles poetry/uv mutual exclusion
- [ ] Rust template sets channel (stable/nightly)
- [ ] Security defaults always present: `dotenv.enable = false`, `unsetEnvVars`
- [ ] Rendered Nix passes `nix-instantiate --parse`
- [ ] Multi-language combinations render correctly (Go+TypeScript, Python+Rust)

**Research Citations:**

### Unit 2.4: .envrc Generation & Native Activation

**Description:** Generate .envrc for direnv users and provide native activation guidance for devenv 2.0+ users.

**Context:** Two activation modes: direnv (`.envrc` with `use devenv`) and native (`devenv hook` in shell rc). The wizard asks which mode. .envrc is machine-owned and idempotent. Validation confirmed native activation replaces direnv for 2.0+ users.

**Desired Outcome:** Generate .envrc when direnv mode selected, or print shell-specific native activation instructions otherwise.

**Steps:**
1. Create `templates/envrc.tmpl` — standard `eval "$(devenv direnvrc)"` + `use devenv`.
2. Implement `generateEnvrc(answers WizardAnswers) *GeneratedFile` — returns nil when native activation selected.
3. Implement `nativeActivationInstructions(shell string) string` — returns instructions for bash/zsh/fish.
4. Include in post-generation output: either "Run `direnv allow`" or "Add `eval \"$(devenv hook <shell>)\"` to your shell rc".

**Acceptance Criteria:**
- [ ] Generated .envrc passes `bash -n` validation
- [ ] .envrc is only generated when direnv mode is selected
- [ ] Native activation instructions cover bash, zsh, fish
- [ ] Post-generation output includes correct next-step instructions

**Research Citations:**
- `research-spikes/gdev-extension-design/devenv-addon-design.md § .envrc` — envrc template
- Validation: native activation confirmed as 2.0+ primary path

**Status:** Not Started

### Unit 2.5: devenv CLI Commands

**Description:** Register the four devenv CLI commands with Cobra: `qsdev devenv init`, `qsdev devenv update`, `qsdev devenv add-service`, `qsdev devenv add-language`.

**Context:** Commands are the user-facing interface for the devenv addon when used standalone (without the devinit orchestrator). Each command must support non-interactive flags and integrate with the generation pipeline.

**Desired Outcome:** All four commands registered, callable, and producing correct output.

**Steps:**
1. Create `addons/devenv/commands.go` — register parent `devenv` command and four sub-commands.
2. `qsdev devenv init` — run wizard or accept flags, generate all files.
3. `qsdev devenv update` — load saved config, regenerate files.
4. `qsdev devenv add-service <name>` — add service to config, regenerate.
5. `qsdev devenv add-language <name>` — add language to config, regenerate.
6. All commands: support `--yes` (skip confirmation), `--force` (overwrite), `--dry-run` (preview only).
7. Wire flag definitions matching devenv-addon-design.md CLI flag spec.

**Acceptance Criteria:**
- [ ] `qsdev devenv init --lang go --yes` generates files without wizard
- [ ] `qsdev devenv update` regenerates from saved config
- [ ] `qsdev devenv add-service postgres` adds PostgreSQL to config
- [ ] `qsdev devenv add-language typescript` adds TypeScript to config
- [ ] All commands support `--dry-run` for preview
