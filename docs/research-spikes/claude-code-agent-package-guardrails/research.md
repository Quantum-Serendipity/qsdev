# Research Summary: Claude Code Agent Package Guardrails

## Overview

Investigate Claude Code's available mechanisms — hooks, skills, MCP servers, permission settings, CLAUDE.md instructions, and custom tooling — for preventing AI coding agents from installing compromised, vulnerable, or supply-chain-attacked packages. The focus is on configure-once defenses that operate unobtrusively in the background: agents should be unable to install known-bad packages without requiring manual review of every install command. This includes integration with vulnerability databases, package provenance checks, lockfile validation, and any other guardrails that can be wired into the Claude Code workflow.

Related spikes:
- `package-supply-chain-security/` — general package manager defenses (feeds into this spike's "what to check" layer)
- `devenv-security/` — devenv.sh-specific boilerplate (feeds into this spike's Nix/devenv integration)

## Topics

### MCP Server Integration for Package Security — Complete
Two production MCP servers exist for package security: Socket.dev (free public server at `mcp.socket.dev`, supply chain scoring via `depscore` tool, supports npm/PyPI/Cargo) and Snyk (bundled with CLI v1.1298.0+, comprehensive scanning via `snyk_sca_scan` and 7 other tools, requires account). Building a custom MCP server that wraps package installation with pre-flight vulnerability checks is fully feasible using the TypeScript SDK (`@modelcontextprotocol/sdk`) and OSV.dev's free, unlimited API. The critical finding is that MCP servers alone cannot enforce security — agents can bypass them via raw Bash — so they must be combined with permission deny rules and PreToolUse hooks for a complete defense. Claude Code's `"type": "mcp_tool"` hook allows PreToolUse hooks to delegate validation to an MCP server, but fails-open on error, requiring a command-hook wrapper for fail-closed behavior. Detailed report: [`mcp-server-research.md`](mcp-server-research.md)

### Claude Code Hooks Mechanism — Complete
Claude Code's PreToolUse hooks are the primary enforcement mechanism for package install guardrails. They fire before every tool call, receive the full command string as structured JSON (`tool_input.command`), and can block execution via exit code 2 or structured JSON with `permissionDecision: "deny"`. Critically, PreToolUse hooks fire *before* permission-mode checks and cannot be bypassed even by `--dangerously-skip-permissions` or `bypassPermissions` mode. Five hook handler types are available (command, http, mcp_tool, prompt, agent), with `command` being the most common for local validation scripts and `http` for team-wide policy enforcement. The `updatedInput` field enables rewriting commands to safer package versions. Enterprise managed settings (`/etc/claude-code/managed-settings.json` on Linux) with `allowManagedHooksOnly: true` make hooks mandatory and non-overridable. Eight bypass vectors were identified (indirect installation via scripts, obfuscated commands, direct manifest editing, disableAllHooks, package aliasing, post-install scripts, subagent patterns, bare `npm install`), each with documented mitigations. The attach-guard plugin is the first purpose-built package guardrail, using Socket.dev API for supply chain scoring and `updatedInput` for version rewriting. Detailed report: [`hooks-research.md`](hooks-research.md)

### Permission Settings & Command Allowlists — Complete
Claude Code's permission system uses three rule arrays (`allow`, `ask`, `deny`) with strict deny-first evaluation order. Rules match Bash command strings via glob patterns (`*` matches any characters including spaces), with shell-operator-aware compound command decomposition (each subcommand in `&&`/`||`/`;`/`|` chains is matched independently). The settings hierarchy has five tiers: managed (non-overridable, deployed via `/etc/claude-code/` on Linux) > CLI args > local project > shared project > user. For package guardrails, deny rules block obvious cases (`Bash(npm install *)`, `Bash(pip install *)`, etc.) while allow rules permit approved wrapper scripts (`Bash(./scripts/safe-install *)`). However, deny rules alone are insufficient: they can be bypassed via subprocess spawning (`python -c "..."`), shell wrapping (`bash -c "npm install evil"`), variable expansion, and `env`/`command` builtins. Two historical vulnerabilities confirmed this: a 50-subcommand bypass (patched v2.1.90) silently disabled deny enforcement on long command chains, and complete deny-rule non-enforcement in v1.0.93. The critical asymmetry between hooks and rules is that hooks can block what rules allow (exit code 2 stops execution before rules are evaluated), but hooks cannot allow what rules deny (hook "allow" only skips prompts, not deny rules). The recommended architecture is three-layer defense: deny rules (fast, simple, catch the obvious), PreToolUse hooks (programmatic, regex-capable, can call external APIs), and OS sandboxing (restricts what subprocesses actually do regardless of how they were invoked). For mandatory non-overridable enforcement, `allowManagedPermissionRulesOnly: true` in managed settings prevents all user/project permission changes, and `disableBypassPermissionsMode: "disable"` blocks the most dangerous mode. Detailed report: [`permissions-research.md`](permissions-research.md)

### CLAUDE.md Instruction-Based Guardrails — Complete
CLAUDE.md instructions are explicitly advisory — delivered as a user message after the system prompt with "no guarantee of strict compliance" per official documentation. Effectiveness is high (~90%+) for specific, concise instructions under normal conditions but degrades under context window pressure, competing instructions, prompt injection, and long sessions. The most critical gap is subagent context loss: custom subagents receive their own system prompt (not CLAUDE.md), and built-in subagents have had CLAUDE.md stripped since v2.1.84 (open regression issue #40459) — meaning subagent-initiated package installs are protected only by hooks and permissions, not behavioral guidance. The primary value of CLAUDE.md for package security is as first-line guidance that reduces enforcement friction: it routes most installs through safe paths voluntarily so deterministic enforcement (hooks, deny rules) only catches edge cases. Best patterns include routing installs through MCP tools/skills, explaining hook behavior so the agent doesn't work around blocks, and establishing approval gates for new dependencies. Detailed report: [`claude-md-guardrails-research.md`](claude-md-guardrails-research.md)

### Custom Skills for Secure Package Installation — Complete
Claude Code skills are prompt-based workflow packages (SKILL.md in `.claude/skills/`) that can wrap package installation with security checks, but they are fundamentally advisory — they cannot prevent agents from bypassing them via raw Bash commands. Skills gain enforcement value only when combined with permission deny rules (blocking raw install commands) and PreToolUse hooks (catching bypass attempts), making the skill the only viable installation path. Skills bring unique capabilities that hooks and MCP servers lack: lifecycle-scoped hooks in SKILL.md frontmatter (active only during skill execution), dynamic context injection, structured check-decide-install-audit pipelines, and user-facing discoverability via `/slash-command` invocation. Claude can auto-invoke skills based on description matching (`disable-model-invocation: false`), but this is probabilistic, not deterministic. The Security Phoenix project (`security-skills-claude-code`) is the most complete real-world example, combining skills with embedded hooks for package install gating. The community has converged on hooks as the primary enforcement mechanism; skills add unique value as the workflow orchestration layer. Detailed report: [`custom-skills-research.md`](custom-skills-research.md)

### Vulnerability Databases & Provenance APIs — Complete
Surveyed 12 APIs and tools for real-time package security validation. OSV.dev is the clear primary choice for hook integration — free, no auth, no rate limits, ~120ms median latency, 60+ ecosystems, covering CVEs via aggregation from GHSA, NVD, and ecosystem databases. Socket.dev's public MCP server is the best supplementary source for supply chain threats beyond CVEs (typosquatting, malware, install scripts, maintenance signals). deps.dev uniquely provides `GetSimilarlyNamedPackages` for typosquatting detection and OpenSSF Scorecards for project health scoring. Ecosystem audit tools (npm audit, pip-audit, cargo-audit) are project-level and too slow (1-5s) for per-package hook integration. Provenance verification (npm Sigstore, PyPI PEP 740) is promising but not yet mature enough for automated enforcement — absence of provenance is not actionable. Significant gap: no Nix-specific vulnerability tool has API surface suitable for real-time hook calls (vulnix is CLI-only system scanner, nix-security-tracker has no public API). Detailed report: [`vulnerability-apis-research.md`](vulnerability-apis-research.md)

### Sibling Spike Cross-Reference — Complete
Cross-referenced all completed reports from `package-supply-chain-security/` (7 reports, 102 sources) and `devenv-security/` (6 reports, 88 sources) to extract findings directly wirable into Claude Code guardrails. Organized into 7 defense categories: publication age quarantine gates (92% of PyPI malware caught within 24 hours; native age-gating now in npm, pnpm, Yarn, pip, uv, Cargo), install script sandboxing (pnpm v10+ blocks scripts by default; npm `--ignore-scripts` + `@lavamoat/allow-scripts`), lockfile enforcement, provenance verification, private registries, organizational scanning, and Nix-specific protections. Each category mapped to all five enforcement layers (PreToolUse hooks, permission deny rules, MCP servers, CLAUDE.md instructions, OS/environment config). Highest-impact immediate actions: environment-level safety defaults in `.npmrc`/`pip.conf`/`nix.conf`, PreToolUse command rewriting via `updatedInput` to append safety flags, permission deny rules for dangerous Nix commands, and Socket.dev MCP integration for pre-install supply chain scoring. Detailed report: [`sibling-spike-cross-reference.md`](sibling-spike-cross-reference.md)

## Conclusions

### Primary Finding
No single Claude Code mechanism is sufficient to prevent agents from installing compromised packages. The required architecture is a **five-layer defense** with three deterministic enforcement layers and two probabilistic advisory layers. The critical insight is that each layer compensates for the others' blind spots:

- **PreToolUse hooks** are the strongest enforcement — deterministic, pre-execution, un-bypassable via permission modes — but can be evaded by obfuscated commands, variable expansion, or indirect installation patterns.
- **Permission deny rules** provide fast, zero-latency blocking of known-dangerous patterns but are bypassable via shell wrappers (`bash -c`), environment prefixing (`env`), and subprocess spawning. Two historical enforcement bugs (v1.0.93 non-enforcement, v2.1.90 50-subcommand bypass) demonstrate they cannot be the sole defense.
- **OS/environment configuration** (`.npmrc ignore-scripts`, `pip.conf only-binary`, `nix.conf sandbox`) works even when hooks and rules are bypassed, but only covers configured package managers and can potentially be modified by the agent.
- **CLAUDE.md instructions** route ~90%+ of installs through safe paths under normal conditions but degrade under context pressure and are not inherited by subagents (regression #40459).
- **Custom skills** provide workflow orchestration and user-facing interfaces but are advisory only — enforcement comes from the deny rules and hooks that make the skill the only viable installation path.

### Key Technical Findings
1. **PreToolUse hooks fire before permission checks** and cannot be bypassed even by `--dangerously-skip-permissions`. This makes them the only truly non-negotiable enforcement point.
2. **OSV.dev is the optimal vulnerability API** for hook integration: free, no auth, no rate limits, ~120ms median latency, 60+ ecosystems. Socket.dev supplements with supply chain risk scoring (typosquatting, malware, install scripts) that CVE databases don't cover.
3. **Publication age gating** is the single highest-impact, lowest-cost defense: 92% of PyPI malware is caught within 24 hours. Native support now exists in npm, pnpm, Yarn, pip, uv, and Cargo.
4. **The `updatedInput` field** in hook responses enables transparent command rewriting — appending `--ignore-scripts`, `--only-binary :all:`, or `--locked` without the agent's awareness.
5. **Enterprise managed settings** (`/etc/claude-code/managed-settings.json` on Linux) with `allowManagedHooksOnly: true` make hooks mandatory and non-overridable, enabling organization-wide enforcement.
6. **Nix vulnerability tooling has a significant gap**: no Nix-specific tool provides API surface suitable for real-time hook integration. vulnix is CLI-only, nix-security-tracker has no public API.

### Residual Risk
Even with all five layers active, three bypass vectors remain partially mitigated: (1) agent editing `package.json`/`Cargo.toml` directly then running bare `npm install` (mitigatable via PostToolUse hooks monitoring manifest changes), (2) heavily obfuscated commands using variable expansion and eval (mitigatable via OS sandbox), and (3) indirect installation via build tools (Make, Gradle) that invoke package managers internally (mitigatable via build tool configuration). Full bypass assessment in [`unified-architecture.md`](unified-architecture.md) Section 9.
