# Phase 16.1: Structured Logging, Error Reporting & Bug Feedback

## Goal

qsdev currently has zero logging infrastructure — only raw `fmt.Fprintf` to stdout/stderr. When a beta tester or early adopter hits a bug, there is no diagnostic trail: no structured logs, no debug output, no way to see what the tool was doing before it failed. The single error message (wrapped via `fmt.Errorf %w`) tells what failed but not what happened leading up to the failure.

This phase adds three capabilities: (1) two-tier structured logging with `log/slog` (stdlib) that writes privacy-safe JSONL files to project-local `.qsdev/logs/` and global `~/.qsdev/logs/` directories, (2) a `--debug` global flag and `QSDEV_LOG` env var for controlling verbosity, and (3) a `qsdev report bug` wizard that walks users through filing GitHub Issues with optional time-windowed log attachment and external tool log ingestion. Design principles: no telemetry or phone-home (all logs are local, user explicitly chooses what to share); privacy-safe by construction via a `RedactingHandler` wrapping `slog.JSONHandler`; always-on at INFO level so logs exist when bugs happen without needing reproduction; user-auditable JSONL format; two-tier separation so project-scoped and global operations never intermingle.

## Dependencies

Phase 5 complete (pre-commit hooks — enterShell and security defaults provide the env var list to extract). Phase 9 complete (system detection — `sysinfo.DetectOS()` provides environment collection for bug reports). Phase 10 complete (self-update — `selfupdate.BackgroundCheck` and `cmd.Main()` provide the inlining target). Phase 12 complete (tool lifecycle — `qsdev logs` and `qsdev report bug` are registered as top-level commands). Phase 13 complete (project config — `.qsdev.yaml` and `.gdev/state.yaml` provide project context for bug reports). Phase 15 complete (health/status — `info.CollectInfo()` provides project metadata for environment collection).

- `--debug` persistent flag and `QSDEV_LOG` env var wired via inlined `cmd.Main()` in `cmd/qsdev/main.go`
- `qsdev logs list/show/path/clean` commands with two-tier awareness (project vs global)

- `slog.Info/Debug` instrumentation across addons and internal packages (~18 files)

### Unit B: Structured Logging Package

**Description:** Create the `internal/logging/` package implementing a `RedactingHandler` wrapping `slog.JSONHandler`, session lifecycle management, two-tier log directory resolution (project-scoped `.qsdev/logs/` vs global `~/.qsdev/logs/`), and count-based log rotation — all using `log/slog` from the Go stdlib.

**Code-Grounded Implementation Note:** The global log directory at `~/.qsdev/` is consistent with the existing self-update cache convention at `selfupdate.go:47`. The `QSDEV_LOG` env var follows Terraform's `TF_LOG` pattern. Session IDs use 6-char hex from `crypto/rand`. Log files are named `qsdev-{ISO8601-safe-timestamp}-{session-id}.jsonl` for collision avoidance and sortability.

**Context:** The logging package is the core infrastructure that all other units in this phase depend on. The `RedactingHandler` intercepts at the `slog.Record` level, cloning each record and walking all `Attr` values through the `Redactor` before forwarding to the inner `slog.JSONHandler`. This ensures secrets never reach disk, even if a developer accidentally passes a token as a log attribute. The two-tier directory design prevents cross-project log intermingling — a developer working across 5 client projects gets isolated per-project log history, making log browsing and bug reporting naturally scoped.

**Desired Outcome:** Calling `logging.Init()` in `PersistentPreRunE` sets up `slog.SetDefault()` with a redacting JSONL handler that writes to the correct tier directory. Sessions are opened/closed cleanly, and old logs are pruned automatically.

**Steps:**
1. Create `internal/logging/redact.go` with `Redactor` struct containing compiled regex patterns for secret value scrubbing (AWS access key IDs `AKIA[A-Z0-9]{16}`, GitHub PATs `gh[pousr]_[A-Za-z0-9_]{36,}`, GitHub fine-grained `github_pat_[A-Za-z0-9_]{22,}`, GitLab PATs `glpat-[A-Za-z0-9\-_]{20,}`, Stripe keys `sk_(live|test)_[A-Za-z0-9]{24,}`, npm tokens `npm_[A-Za-z0-9]{36}`, JWTs `eyJ...`, private key headers, URLs with embedded credentials, generic token patterns after known keys).
2. Add key-name deny list to `Redactor` using `secrets.SensitiveKeyPatterns` with word-boundary matching so `"tokenizer_count"` is NOT redacted but `"access_token"` is.
3. Add env-name deny list to `Redactor` using `secrets.KnownCredentialVars` to redact any attribute whose key matches a known credential env var name.
4. Create `internal/logging/handler.go` with `RedactingHandler` implementing `slog.Handler` — intercepts `Handle()`, clones the record, walks all attrs through the `Redactor`, forwards scrubbed record to the inner handler. Add `TeeHandler` that fans out to multiple handlers (file + stderr in `--debug` mode).
5. Create `internal/logging/session.go` with `Session` struct (ID, Command, StartedAt, logFile). `Init(Config)` creates the log dir, opens the session file, configures `slog.SetDefault()`, writes an opening record with version/OS/arch metadata, and runs `cleanOldLogs()`. `Close()` writes a closing record with duration, flushes, and closes the file.
6. Create `internal/logging/paths.go` with `GlobalLogDir()` (returns `~/.qsdev/logs/`, overridable via `QSDEV_LOG_DIR`), `ProjectLogDir(projectRoot)` (returns `{projectRoot}/.qsdev/logs/`), `ResolveLogDir(projectRoot, isProjectScoped)` (routes to project tier or global tier), and `IsProjectScopedCommand(cmdPath)` (classifies commands as project-scoped or global).
7. Define tier routing: commands under `init`, `status`, `check`, `repair`, `enable`, `disable`, `update`, `teardown`, `info`, `outdated`, `evidence`, `team-report`, `config`, `devenv`, `claude` route to project tier; commands `self-update`, `version`, `report`, `logs --global`, `completion` route to global tier; project-scoped commands outside a qsdev project fall back to global tier.
8. Create `internal/logging/rotate.go` with count-based pruning — on `Init()`, scan the active log dir, delete `.jsonl` files beyond 50 (oldest first by filename timestamp). Add `CleanOldLogs(dir, maxAge)` for age-based cleanup. Rotation runs independently per tier.
9. Implement `LevelFromEnv()` that reads `QSDEV_LOG` env var (`debug`, `info`, `warn`, `error`, `off`; default: `info`).
10. Write comprehensive tests: `redact_test.go` with table-driven tests for every secret pattern (positive: AWS key, GH token, JWT, URL with creds all become `[REDACTED]`; negative: `"tokenizer_count"` and `"mypassword_field"` NOT redacted due to word boundaries; nested grouped attrs with secrets inside are redacted). `handler_test.go`, `session_test.go`, `paths_test.go`, `rotate_test.go` with unit tests for each component.

**Acceptance Criteria:**
- [ ] `RedactingHandler` scrubs all 10+ secret patterns before any log line reaches disk
- [ ] Word-boundary matching prevents false positives (`"tokenizer"` kept, `"token"` redacted, `"access_token"` redacted)
- [ ] `TeeHandler` fans out to both file and stderr when `--debug` is active
- [ ] `Session.Init()` creates the correct tier directory, opens a JSONL file, and writes an opening record
- [ ] `Session.Close()` writes a closing record with duration and flushes the file
- [ ] `ResolveLogDir()` returns project tier for project-scoped commands, global tier otherwise
- [ ] Project-scoped commands outside a qsdev project fall back to global tier
- [ ] Count-based rotation keeps at most 50 log files per tier directory
- [ ] `LevelFromEnv()` correctly parses `QSDEV_LOG` values including `off` (disables logging)
- [ ] All tests pass: `command go test ./internal/logging/...`

### Unit G: External Log Ingestion Core

**Description:** Create the `internal/extlog/` package with core types (`LogEntry`, `LogFile`, `LogLevel`, `TimestampSource`), a self-registering provider `Registry`, a two-pass external log scrubber (tool-specific + universal), ANSI escape code stripping, and error-centric truncation for large log files.

**Code-Grounded Implementation Note:** The provider registry follows the self-registering `init()` pattern from `internal/ecosystem/registry.go` — each provider package has an `init()` function that calls `extlog.RegisterProvider()`. The `LogProvider` interface has five methods: `Name()`, `DisplayName()`, `Detect()`, `Discover()`, and `Parse()`. `TimestampSource` distinguishes parsed timestamps (line-level precision) from carried timestamps (propagated from previous line) from mtime timestamps (file-level precision only), letting consumers know how much to trust time correlation.

**Context:** When a user hits a bug, the root cause often lies in an external tool's behavior (npm install failed, nix build errored, devenv shell crashed), not in qsdev itself. Most tools do NOT write persistent log files — they write to stderr and it's gone. Only npm (`~/.npm/_logs/`), nix (`/nix/var/log/nix/drvs/`), and pip (if configured) persist logs reliably. This means the system needs a hybrid strategy: lazy scan for persistent logs + eager capture of ephemeral stderr (Unit I). External logs receive more aggressive scrubbing than qsdev's own logs because we don't control what tools emit.

**Desired Outcome:** A pluggable provider framework where adding a new tool's log support requires implementing one interface and adding one `init()` registration — no changes to core code.

**Steps:**
1. Create `internal/extlog/extlog.go` with core types: `LogEntry` (Timestamp, TimestampSource, Level, Source, Message, File, LineNumber), `LogFile` (Path, Provider, ModTime, Size), `LogLevel` enum (debug, info, warn, error, unknown), `TimestampSource` enum (parsed, carried, mtime).
2. Define `LogProvider` interface: `Name() string`, `DisplayName() string`, `Detect(projectRoot, homeDir string) bool`, `Discover(projectRoot, homeDir string, since time.Time) ([]LogFile, error)`, `Parse(r io.Reader, sourceFile string) ([]LogEntry, error)`.
3. Create `internal/extlog/registry.go` with thread-safe `Registry` struct, `RegisterProvider(LogProvider)` function, and `DetectAll(projectRoot, homeDir string) []LogProvider` method that returns providers with available logs. Providers register independently of ecosystem detection.
4. Create `internal/extlog/scrubber.go` implementing the two-pass scrubbing pipeline: Pass 1 (tool-specific) is handled by each provider's `Parse()` method; Pass 2 (universal) applies all regex patterns from the `Redactor` (Unit B), normalizes paths (`/home/username/...` to `~/...`, `/path/to/project/...` to `./...`), scrubs private hostnames (internal registry/cache URLs to `[REDACTED_HOST]` while preserving public hosts: npmjs.org, pypi.org, crates.io, github.com).
5. Create `internal/extlog/ansi.go` with ANSI escape code stripping function using pattern `\x1b\[[0-9;]*m` — applied to all external log content.
6. Create `internal/extlog/truncate.go` with error-centric truncation: find all ERROR/FATAL entries, include 5 lines of context before and 3 after each error cluster, fall back to head (first 20 lines) + tail (last 50 lines) if no errors found. Accept a size budget parameter (default 8KB per provider).
7. Create `internal/extlog/collect.go` with orchestration function `CollectExternalLogs(projectRoot, homeDir string, since time.Time, providers []LogProvider)` that discovers, collects, scrubs, normalizes, and time-windows logs from all provided providers.
8. Write comprehensive tests: `scrubber_test.go` (npm auth tokens, nix access tokens, path normalization, private hostname scrubbing, public host preservation), `ansi_test.go` (various ANSI sequences stripped), `truncate_test.go` (error-centric windowing, head+tail fallback, size budget enforcement), `collect_test.go` (multi-provider orchestration).

**Acceptance Criteria:**
- [ ] `LogProvider` interface is implementable with five methods
- [ ] `Registry` supports self-registration via `init()` pattern
- [ ] `DetectAll()` returns only providers that found available logs
- [ ] Two-pass scrubber: tool-specific patterns in Pass 1, universal patterns in Pass 2
- [ ] Path normalization: `/home/username/...` becomes `~/...`, project paths become `./...`
- [ ] Private hostnames scrubbed to `[REDACTED_HOST]`; public hosts (npmjs.org, pypi.org, crates.io, github.com) preserved
- [ ] ANSI escape codes fully stripped from all external log content
- [ ] Error-centric truncation includes 5 lines before and 3 after each error cluster
- [ ] Head+tail fallback (20+50 lines) when no errors found
- [ ] Size budget enforced per provider (default 8KB)
- [ ] `internal/extlog/` does not import any addon package (depguard compliant)

### Unit I: Ephemeral Output Capture

**Description:** Create a `CaptureWriter` in `internal/extlog/capture/` that tees external tool output to both the original destination (user's terminal) and a capture file in `.qsdev/logs/capture/`, then wire it into existing `exec.Command` call sites across the codebase.

**Context:** Most external tools write to stderr and the output is gone after the process exits. When these tools fail, the user sees the error message scroll past on their terminal but has no record of it. `CaptureWriter` solves this by transparently tee-ing output to a capture file that the devenv log provider (Unit H) can later discover and the bug report wizard (Unit E/J) can include. Capture files follow the same two-tier routing and rotation as regular logs.

**Desired Outcome:** After wiring `CaptureWriter`, key external tool invocations produce capture files in `.qsdev/logs/capture/` that are automatically discovered by the devenv log provider and available for bug report attachment.

3. Add capture file location logic following two-tier routing: `.qsdev/logs/capture/` for project-scoped commands, `~/.qsdev/logs/capture/` for global commands.
