# devenv.sh Integration Surface Area for a gdev Addon

## Executive Summary

devenv.sh is a Nix-based developer environment tool that manages its configuration through two primary files: `devenv.yaml` (dependency/input management, YAML) and `devenv.nix` (environment configuration, Nix language). A gdev addon that sets up devenv.sh environments must generate and manage both files, plus the optional `.envrc` for direnv integration and `.gitignore` entries. The configuration surface area is large — 60+ language modules, 45+ services, processes, tasks, scripts, git-hooks, containers, outputs, env vars, and shell hooks — but follows consistent, predictable patterns that make templated generation feasible.

## 1. Configuration Model Architecture

### File Inventory

devenv.sh uses a small set of files, each with a distinct role:

| File | Format | Role | Generated by `devenv init` |
|------|--------|------|----------------------------|
| `devenv.yaml` | YAML | Input/dependency management, imports, nixpkgs config, global settings | Yes |
| `devenv.nix` | Nix | Environment configuration (packages, languages, services, scripts, etc.) | Yes |
| `devenv.lock` | JSON | Pinned input revisions (like package-lock.json) | Auto-generated |
| `.gitignore` | Text | Excludes `.devenv`, `.pre-commit-config.yaml` | Yes |
| `.envrc` | Bash | direnv integration (optional) | **No** — must be created manually |
| `flake.nix` | Nix | Alternative: Nix Flakes integration (instead of devenv.yaml) | Only with `nix flake init --template` |
| `flake.lock` | JSON | Flake input pins (only in flake mode) | Auto-generated |

**Key insight for gdev**: The addon needs to generate at minimum `devenv.yaml` and `devenv.nix`. The `.envrc` is optional but highly recommended for teams using direnv. The `devenv.lock` is auto-generated and should not be templated.

### Two Modes of Operation

devenv.sh supports two distinct integration modes:

1. **Native mode** (recommended): Uses `devenv.yaml` + `devenv.nix`. The `devenv` CLI manages inputs, evaluation, and caching directly. This mode gets the full feature set: evaluation caching, container support, GC protection, secretspec, polyrepo references.

2. **Flake mode**: Uses `flake.nix` + `devenv.nix` modules passed to `devenv.lib.mkShell`. Works within existing Nix flake ecosystems. Loses some features: no evaluation caching, no built-in container support, no secretspec, tests cannot start processes.

**Recommendation for gdev**: Target native mode. It is simpler, more featureful, and the recommended path. Flake mode is for teams already invested in flakes.

### devenv.yaml Structure

The YAML file controls:

```yaml
# Inputs — Nix dependency sources
inputs:
  nixpkgs:
    url: github:cachix/devenv-nixpkgs/rolling
  git-hooks:
    url: github:cachix/git-hooks.nix
  # Custom inputs for additional package sources

# Imports — composable environment modules
imports:
  - ./frontend
  - ./backend
  - some-input/path/to/module

# Nixpkgs configuration
nixpkgs:
  allow_unfree: false
  allow_broken: false
  cuda_support: false
  rocm_support: false
  permitted_insecure_packages: []
  permitted_unfree_packages: []

# Shell behavior
clean:
  enabled: false
  keep: []
impure: false
reload: true          # auto-reload on file changes (2.0+)
strict_ports: false   # fail vs auto-increment on port conflicts

# Profiles
profile: ""           # default profile name

# Secrets
secretspec:
  enable: false
  provider: ""
  profile: ""

# Version constraint
require_version: ">= 2.0"
```

### devenv.nix Structure

The Nix file is a function returning an attribute set:

```nix
{ pkgs, lib, config, inputs, ... }:

{
  # --- Packages ---
  packages = [ pkgs.git pkgs.curl ];

  # --- Environment Variables ---
  env.DATABASE_URL = "postgres://localhost/mydb";
  env.NODE_ENV = "development";

  # --- Languages ---
  languages.go.enable = true;
  languages.javascript.enable = true;

  # --- Scripts ---
  scripts.migrate.exec = "...";

  # --- Processes ---
  processes.server.exec = "...";

  # --- Services ---
  services.postgres.enable = true;

  # --- Tasks ---
  tasks."myapp:build".exec = "...";

  # --- Git Hooks ---
  git-hooks.hooks.shellcheck.enable = true;

  # --- Shell Initialization ---
  enterShell = ''...'';

  # --- Testing ---
  enterTest = ''...'';

  # --- Containers ---
  containers.shell = { ... };

  # --- Outputs ---
  outputs.myapp = config.languages.rust.import ./. {};
}
```

## 2. Configuration Knobs — Complete Surface Area

### 2.1 Packages

The simplest configuration dimension. A flat list of Nix packages:

```nix
packages = [ pkgs.git pkgs.curl pkgs.jq pkgs.ripgrep ];
```

Can also use `inputsFrom` to inherit packages from other derivations. Over 100,000 packages available from nixpkgs.

**gdev surface**: List of package names. The addon could offer common package presets (e.g., "web dev basics": git, curl, jq, httpie).

### 2.2 Languages (60+ supported)

Each language follows a consistent pattern:

```nix
languages.<name>.enable = true;          # Required: activate the language
languages.<name>.package = pkgs.<pkg>;   # Optional: override package
languages.<name>.version = "x.y.z";     # Optional: pin version (uses overlays)
languages.<name>.lsp.enable = true;      # Optional: language server (default: true)
languages.<name>.lsp.package = pkgs.<x>; # Optional: override LSP package
```

**Language-specific extras vary significantly by ecosystem:**

- **Go**: `delve.enable`, `enableHardeningWorkaround`, auto-sets `GOROOT`/`GOPATH`/`GOTOOLCHAIN`
- **JavaScript**: Package manager selection (`npm`, `pnpm`, `yarn`, `bun` — each with `.enable` and `.install.enable`), `corepack.enable`, `directory`
- **Python**: `venv.enable`, `venv.requirements`, `poetry.*` (15+ options), `uv.*` (10+ options), `libraries`, `manylinux.enable` — Poetry and uv are mutually exclusive
- **Rust**: `channel` (nixpkgs/stable/beta/nightly), `components`, `targets`, `toolchainFile`, linker selection (`lld`/`mold`/`wild`), `cranelift.enable`, `rustflags`
- **TypeScript**: Minimal — just `enable` and `lsp.*`

**Pattern for gdev**: The addon needs a language selector (which languages to enable), then language-specific sub-questions (e.g., "Which Python package manager?" "Which Node.js package manager?"). The `enable = true` + optional version/package pattern is universal, but the extras are language-specific.

### 2.3 Services (45+ supported)

Higher-level abstraction over processes with pre-configured defaults:

```nix
services.postgres = {
  enable = true;
  package = pkgs.postgresql_15;
  initialDatabases = [{ name = "mydb"; }];
  extensions = extensions: [ extensions.postgis ];
  settings.shared_preload_libraries = "timescaledb";
  initialScript = "CREATE EXTENSION IF NOT EXISTS timescaledb;";
};
```

Available services include: PostgreSQL, MySQL, MongoDB, Redis, Memcached, Elasticsearch, Cassandra, CouchDB, Kafka, RabbitMQ, NATS, Nginx, Caddy, Vault, Keycloak, Prometheus, and 25+ more.

Services launch via `devenv up`, state persists in `$DEVENV_STATE`, and each service has its own module with service-specific options.

**gdev surface**: Service selection (which services to enable), with service-specific configuration (e.g., "PostgreSQL version? Initial database name? Extensions?"). Most teams need 1-3 services.

### 2.4 Processes

Custom long-running processes with supervision:

```nix
processes.server = {
  exec = "python -m http.server 8080";
  # devenv 2.0+ features:
  restart = { policy = "on_failure"; max = 5; };
  ready = { http.get = { port = 8080; path = "/health"; }; };
  after = [ "db@ready" ];
  watch = { paths = [ "./src" ]; extensions = [ "py" ]; };
  ports.http.allocate = 8080;    # auto-allocation
};
```

Key features:
- Dependency ordering (`after`/`before` with `@started`/`@ready`/`@completed` suffixes)
- Restart policies (`on_failure`/`always`/`never`, max retries)
- Ready probes (exec, HTTP, systemd notify)
- File watching (paths, extensions, ignore patterns)
- Socket activation (TCP, Unix stream)
- Watchdog heartbeat monitoring
- Automatic port allocation (prevents collisions)
- Alternative process managers (process-compose, overmind, honcho, hivemind, mprocs)

**gdev surface**: Most teams use services rather than raw processes. The addon could offer a "custom process" option for non-service workloads.

### 2.5 Scripts

Named shell scripts exposed in the environment:

```nix
scripts.migrate = {
  exec = ''
    cd backend && python manage.py migrate
  '';
  packages = [ pkgs.python3 ];  # script-scoped deps
  description = "Run database migrations";
};

# Multi-language support
scripts.analyze = {
  exec = ''print("analyzing...")'';
  package = config.languages.python.package;  # runs as Python
};
```

Features: argument forwarding (`"$@"`), per-script package scoping, multi-language execution via `package`/`binary`, external file references (`exec = ./script.sh`).

**gdev surface**: Teams commonly need project-specific scripts. The addon could offer common script templates (build, test, lint, migrate, seed) based on project type.

### 2.6 Environment Variables

Simple key-value pairs:

```nix
env.DATABASE_URL = "postgres://localhost/mydb";
env.NODE_ENV = "development";
env.GREET = "hello";
```

Accessible throughout the environment (scripts, processes, enterShell).

**gdev surface**: The addon should support arbitrary env var configuration. Could offer project-type defaults.

### 2.7 Tasks (v1.2+)

DAG-based task execution with caching:

```nix
tasks."myapp:build" = {
  exec = "npm run build";
  before = [ "devenv:enterShell" ];
  after = [ "myapp:codegen@succeeded" ];
  status = "check-if-build-needed";     # skip if status returns 0
  execIfModified = [ "src/**/*.ts" ];   # file-change gating
  input = { key = "value"; };           # JSON inputs
  cwd = "./frontend";
  package = pkgs.nodejs;                # non-bash execution
};
```

Built-in lifecycle events: `devenv:enterShell` (runs before shell entry), `devenv:enterTest` (runs before tests). Processes are auto-registered as tasks with `devenv:processes:` prefix.

**gdev surface**: Tasks are an advanced feature. The addon could offer common task patterns but this is likely a "power user" configuration.

### 2.8 Git Hooks (pre-commit integration)

Via git-hooks.nix integration:

```nix
git-hooks.hooks = {
  # Built-in hooks
  shellcheck.enable = true;
  black.enable = true;           # Python formatter
  rustfmt.enable = true;
  clippy.enable = true;
  prettier.enable = true;        # JS/TS formatter

  # Custom hooks
  my-hook = {
    entry = "my-linter";
    files = "\\.py$";
    types = [ "text" ];
    language = "system";
    pass_filenames = true;
  };
};
```

**Important**: As of devenv 2.0, the `git-hooks` input is no longer included by default. It must be explicitly added to `devenv.yaml`:

```yaml
inputs:
  git-hooks:
    url: github:cachix/git-hooks.nix
```

Hooks auto-install to `.git/hooks/pre-commit` when entering `devenv shell`.

**gdev surface**: Hook selection based on project languages. The addon should auto-add the git-hooks input when any hooks are enabled.

### 2.9 Shell Initialization (`enterShell`)

Bash code executed when entering the environment:

```nix
enterShell = ''
  echo "Welcome to the dev environment!"
  export PATH="$HOME/.local/bin:$PATH"
'';
```

The docs recommend using tasks instead for complex setup operations (better dependency control, parallelism, caching).

**gdev surface**: Simple string, often project-specific. The addon could inject standard setup like PATH additions.

### 2.10 Containers

OCI container generation without Docker:

```nix
containers.myapp = {
  copyToRoot = ./dist;
  startupCommand = config.processes.server.exec;
};
```

Built-in container types: `shell` (interactive env) and `processes` (runs all processes). Supports registry push via skopeo.

**gdev surface**: Container config is opt-in and varies significantly. The addon could offer a "containerize" toggle that generates sensible defaults.

### 2.11 Testing

```nix
enterTest = ''
  wait_for_port 8080
  curl -s localhost:8080 | grep "Hello"
'';
```

`devenv test` builds the environment, starts processes, runs `enterTest`, then tears down. The `config.devenv.isTesting` flag enables conditional config.

### 2.12 Outputs (v1.1+)

Exposes Nix derivations for packaging:

```nix
outputs = {
  myapp = config.languages.rust.import ./. {};
};
```

Language-specific importers: Rust (crate2nix), Python (uv2nix). Custom outputs via `config.lib.types.outputOf`.

### 2.13 Imports (Composition)

```yaml
# devenv.yaml
imports:
  - ./frontend
  - ./backend
  - shared-config/modules/base
```

Each import is a directory containing its own `devenv.nix`. Configurations merge at the root level.

### 2.14 SecretSpec (v1.8+)

Provider-agnostic secrets management:

```yaml
# devenv.yaml
secretspec:
  enable: true
  provider: keyring   # or: dotenv, 1password, env
  profile: dev
```

Declared in `secretspec.toml`, prompted from backends rather than stored in `.env` files.

## 3. Initialization: What `devenv init` Creates

Running `devenv init` in a fresh directory creates:

1. **`devenv.nix`** — minimal starter config:
   ```nix
   { pkgs, ... }:
   {
     packages = [ pkgs.git ];
     enterShell = ''
       hello
     '';
   }
   ```

2. **`devenv.yaml`** — default inputs:
   ```yaml
   inputs:
     nixpkgs:
       url: github:cachix/devenv-nixpkgs/rolling
   ```

3. **`.gitignore`** — standard exclusions:
   ```
   .devenv
   .pre-commit-config.yaml
   ```

Notable: `.envrc` is NOT created by `devenv init`. This is a manual step for direnv users.

For the flake path, `nix flake init --template github:cachix/devenv` creates `flake.nix` and `.envrc` instead.

## 4. Module System Architecture

### How It Works

devenv.sh uses the NixOS module system internally. Every configuration file (`devenv.nix`) is a NixOS-style module: a function taking `{ pkgs, lib, config, ... }` and returning an attribute set of option values.

The top-level module (`top-level.nix`) imports ~19 sub-modules covering:
- Profiles, outputs, files
- Processes, scripts, tasks
- Languages (60+ individual modules)
- Services (45+ individual modules)
- Integrations (direnv, git-hooks, etc.)

### Module Pattern (from go.nix source)

Every language/service module follows this pattern:

```nix
{ pkgs, config, lib, ... }:
let
  cfg = config.languages.go;
in
{
  # 1. Declare options
  options.languages.go = {
    enable = lib.mkEnableOption "tools for Go development";
    package = lib.mkOption { type = lib.types.package; default = pkgs.go; };
    version = lib.mkOption { type = lib.types.nullOr lib.types.str; default = null; };
    # ... more options
  };

  # 2. Define config when enabled
  config = lib.mkIf cfg.enable {
    packages = [ cfg.package ];
    env.GOROOT = cfg.package + "/share/go/";
    enterShell = ''export PATH=$GOPATH/bin:$PATH'';
  };
}
```

Key patterns:
- **`config.lib.getInput`**: Lazy input fetching — only downloads overlays when a version option is actually used
- **`lib.mkIf cfg.enable`**: Entire config block gated on the enable flag
- **`buildWithSpecificGo`**: Package override pattern to ensure tool consistency
- **Option types**: `types.package`, `types.str`, `types.bool`, `types.nullOr`, `types.listOf`, `types.attrsOf`

### Creating Custom Modules

Custom modules can be shared via imports:

```yaml
# devenv.yaml
imports:
  - my-org-config/modules/base-env
```

Where `my-org-config/modules/base-env/devenv.nix` contains team-standard configuration. This is the composition mechanism a gdev addon would leverage.

### Module Inputs Available to devenv.nix

Every `devenv.nix` receives these as function arguments:

- **`pkgs`**: The nixpkgs package set for the current system
- **`lib`**: Nix standard library (data manipulation functions)
- **`config`**: The final resolved configuration (self-referential)
- **`inputs`**: All inputs declared in `devenv.yaml`
- **`...`**: Catch-all for future additions

## 5. Shell Activation Mechanisms

### Option 1: Manual (`devenv shell`)

Enter explicitly with `devenv shell`. Spawns a subshell. Exit with `exit` or Ctrl-D.

### Option 2: Native Auto-Activation (`devenv hook`)

Add to shell config (e.g., `~/.zshrc`):
```
eval "$(devenv hook zsh)"
```

Then `devenv allow` in the project directory. Auto-activates when `cd`-ing into any directory with `devenv.yaml`. Spawns subshell. No external dependencies.

**Limitation**: Only detects projects with `devenv.yaml` — not `devenv.nix` alone.

### Option 3: direnv Integration

Create `.envrc`:
```bash
eval "$(devenv direnvrc)"
use devenv
```

Then `direnv allow`. Modifies current shell in-place (no subshell). Requires direnv installed separately. Can pass options: `use devenv --option services.postgres.enable:bool true`.

### Comparison

| Feature | `devenv shell` | `devenv hook` | direnv |
|---------|---------------|---------------|--------|
| Auto-activation | No | Yes | Yes |
| External deps | None | None | direnv |
| Shell modification | Subshell | Subshell | In-place |
| Per-project config file | None | None | `.envrc` |
| IDE integration | Manual | Manual | Good (many IDEs support direnv) |

**Recommendation for gdev**: Generate `.envrc` by default since direnv has the best IDE integration. Also document the `devenv hook` alternative for teams that prefer no extra dependencies.

## 7. Comparison with Alternatives

### vs. `nix develop` (plain flakes)

| Aspect | devenv.sh | nix develop |
|--------|-----------|-------------|
| Config format | devenv.yaml + devenv.nix | flake.nix |
| Learning curve | Lower (structured options) | Higher (raw Nix) |
| Services | 45+ pre-built | DIY |
| Process management | Built-in (native or process-compose) | None |
| Language support | 60+ with structured options | Manual package lists |
| Evaluation caching | Yes (sub-100ms) | No |
| Container generation | Built-in | External tooling |
| Git hooks | First-class via git-hooks.nix | Manual integration |
| direnv | Built-in `devenv direnvrc` | Via nix-direnv |

**Verdict**: devenv.sh provides a significantly higher-level abstraction. `nix develop` is lower-level but more flexible for unusual configurations.

### vs. flake-parts

flake-parts provides a module system for flakes but does not include pre-built service/language modules. devenv.sh's value is the 60+ language modules, 45+ services, process management, and developer UX tooling (TUI, auto-reload, caching). flake-parts is complementary — you can use devenv within flake-parts.

### Key devenv.sh Limitations

1. **Nix required**: Team members must install Nix. No escape hatch for non-Nix environments.
2. **nixpkgs version lag**: Default `devenv-nixpkgs/rolling` updates monthly. Bleeding-edge packages require adding `nixpkgs-unstable` as a second input.
3. **macOS container limitation**: Container builds require a remote Linux builder on macOS.
4. **Flake mode feature gaps**: No evaluation caching, no container support, tests can't start processes.
5. **`devenv.yaml` import limitation**: Remote inputs cannot yet be imported as `devenv.yaml` files (only `devenv.nix`).
6. **Module complexity**: Language modules with many options (Python: 30+, Rust: 20+) can be overwhelming without guidance.
7. **No Windows native**: WSL2 only.

## 8. Edge Cases and Failure Modes for gdev Addon

These are the tricky scenarios the addon's generated configuration may encounter:

1. **First-run cold cache**: Initial `devenv shell` downloads and builds everything. Can take 5-30 minutes depending on languages and services. Subsequent runs are sub-100ms thanks to evaluation caching. The addon should warn users about first-run time.

2. **Git-hooks input missing**: devenv 2.0 removed the default `git-hooks` input. If the addon enables any git-hooks but forgets to add the input to `devenv.yaml`, evaluation fails. The addon MUST add `git-hooks: { url: github:cachix/git-hooks.nix }` whenever any hook is enabled.

3. **Language version overlay dependencies**: Pinning language versions (e.g., `languages.go.version = "1.22.0"` or `languages.python.version = "3.11"`) requires external overlays/inputs. Go uses `go-overlay` (auto-fetched via `config.lib.getInput`), Python requires explicitly adding `nixpkgs-python` input in `devenv.yaml`. The addon must handle these input dependencies per language.

4. **Mutually exclusive options**: Python's `poetry.install.enable` and `uv.sync.enable` cannot both be true. The addon wizard must enforce this exclusivity.

5. **Service state persistence**: Service data (e.g., PostgreSQL databases) persists in `$DEVENV_STATE`. Changing service configuration (e.g., PostgreSQL version) requires manually deleting the service state directory before restarting. The addon should document this.

6. **Path inputs and .gitignore**: Using `path:` URI inputs copies entire directories to the Nix store without respecting `.gitignore`. The addon should prefer `git+file:` for local paths.

7. **Impure evaluation**: Some packages or configurations need impure evaluation (accessing system state). This requires `impure: true` in `devenv.yaml`. Common trigger: packages that read system GPU drivers, CUDA configurations, etc.

8. **nixpkgs unfree packages**: Packages like CUDA, VSCode, or some fonts are "unfree." The addon must set `nixpkgs.allow_unfree: true` or `nixpkgs.permitted_unfree_packages` when these are needed.

9. **Port conflicts**: By default devenv auto-increments ports when conflicts occur. `strict_ports: true` makes it fail instead. Teams sharing a machine need auto-allocation; CI environments may need strict mode.

10. **direnv trust scope**: direnv trusts per `.envrc` file content. If the addon regenerates `.envrc`, users must re-run `direnv allow`. The native `devenv hook` trusts per directory, avoiding this issue.

11. **Nix string escaping**: Shell variables in `devenv.nix` strings need `''${var}` escaping (double single-quotes + dollar). The addon's Nix code generator must handle this correctly.

12. **macOS SDK handling**: The top-level module has special logic for partitioning packages on macOS for Apple SDK compatibility. The addon doesn't need to handle this directly (devenv does it internally), but should be aware that some packages may not build on macOS without the SDK.

## 9. What the gdev Addon Must Manage

Based on the full analysis, the addon's configuration surface area breaks down into these tiers:

### Tier 1: Must Generate (Every Project)

1. **`devenv.yaml`** — inputs (nixpkgs, optionally git-hooks, language-specific overlays), imports, nixpkgs settings
2. **`devenv.nix`** — the primary environment definition
3. **`.gitignore`** entries — `.devenv`, `.pre-commit-config.yaml`, `.direnv`

### Tier 2: Should Generate (Most Projects)

4. **`.envrc`** — direnv integration for auto-activation
5. **Language configuration** — `languages.<name>.enable` + ecosystem-specific options
6. **Package list** — common tools (git, curl, etc.)

### Tier 3: Optional (Based on Project Needs)

7. **Services** — database, cache, message queue selection and config
8. **Scripts** — project-specific convenience commands
9. **Git hooks** — linter/formatter hooks matching selected languages
10. **Environment variables** — project-specific env configuration
11. **Processes** — custom long-running processes beyond services
12. **Tasks** — DAG-based build/setup tasks
13. **Containers** — OCI container generation config
14. **Outputs** — Nix packaging for deployment
15. **SecretSpec** — secrets management integration
16. **Profiles** — environment variants (dev/staging/prod)

### Wizard Decision Tree

The addon's wizard should walk through roughly this sequence:

1. **Project type** → determines language(s) and common patterns
2. **Language versions** → pins via `version` options or defaults
3. **Package manager** → language-specific (npm/pnpm/yarn/bun, poetry/uv, etc.)
4. **Services needed** → database, cache, queue selection
5. **Git hooks** → auto-suggest based on languages (rustfmt for Rust, prettier for JS, etc.)
6. **direnv** → yes/no for .envrc generation
7. **Extra packages** → common tools checklist
8. **Custom env vars** → key-value pairs

Each step maps to specific sections of `devenv.yaml` and `devenv.nix` that the addon templates.
