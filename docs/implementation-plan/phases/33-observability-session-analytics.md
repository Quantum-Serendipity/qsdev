# Phase 33: Observability & Session Analytics

## Goal

Add a lightweight observability sidecar and privacy-safe session analytics. The observability sidecar (`grafana/otel-lgtm` Docker container) provides a zero-config local Grafana + Prometheus + Loki + Tempo stack via a single Docker image, replacing what would otherwise be a four-service devenv configuration. Session analytics fills the community gap for consulting-safe team analytics: metadata-only event collection that never touches source code, prompt text, or conversation content. A gdev-managed ccusage integration provides immediate cost visibility as the lowest-friction adoption moment.

## Dependencies

Phase 12 complete (tool lifecycle management, `qsdev enable/disable` machinery, file ownership registry). Phase 16 complete (developer experience polish — `qsdev status`, shell integration, enterShell hooks, OTEL env var pattern from Unit 16.7).

## Phase Outputs

- `qsdev enable observability` command deploying `grafana/otel-lgtm` Docker container
- `docker-compose.observability.yml` generated in project root
- `qsdev observability` subcommand group (up/down/status/logs)
- OTEL env var generation in devenv.nix when observability is enabled
- `qsdev enable ccusage` and `qsdev cost` command wrapper
- Metadata-only analytics event engine collecting session/cost/hook metrics only
- `qsdev analytics report` and `qsdev analytics export` commands
- Session replay export via `qsdev session export <session-id>` (opt-in)
- Pre-configured Grafana dashboard for Claude Code analytics when OTel sidecar is active

### Unit 33.1: OTel Sidecar Deployment

**Description:** Implement `qsdev enable observability` to deploy the `grafana/otel-lgtm` Docker container as a project-scoped observability sidecar. The container provides Grafana (port 3000), OTLP gRPC (port 4317), and OTLP HTTP (port 4318) with zero configuration. Generate `docker-compose.observability.yml` in the project root and wire auto-start into devenv's `enterShell` hook.

**Context:** The observability sidecar research established that Grafana, Loki, Tempo, and Prometheus are not available as native devenv.sh services — they require Docker. The `grafana/otel-lgtm` all-in-one image (`github.com/grafana/docker-otel-lgtm`) bundles the entire LGTM stack (Loki + Grafana + Tempo + Mimir/Prometheus) plus an OpenTelemetry Collector into a single container with zero configuration required. A developer runs one command, and a complete local observability stack is available. This replaces what would otherwise be a four-container docker-compose setup with preconfigured dashboards. The image exposes three ports: 3000 (Grafana UI), 4317 (OTLP gRPC), and 4318 (OTLP HTTP). Data persistence is provided via a named Docker volume, preserving Grafana dashboards and collected data across container restarts.

The `qsdev enable observability` pattern follows the existing tool lifecycle system from Phase 12 (Unit 12.1): it registers the sidecar as a managed tool, generates files with section markers, and provides the standard `qsdev observability up/down/status/logs` subcommand group. ### Unit 33.2: OTEL Environment Variable Generation

**Description:** Generate OpenTelemetry standard environment variables into devenv.nix when observability is enabled. These env vars configure any OTel-instrumented application to export telemetry to the local sidecar automatically, with no application code changes required.

**Context:** OpenTelemetry's environment variable specification is the standard configuration mechanism — every OTel SDK (Go, Python, Node.js, Java, Rust) reads `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_SERVICE_NAME`, and `OTEL_RESOURCE_ATTRIBUTES` from the environment. When a developer opens their devenv shell, these vars are already set, meaning any OTel-instrumented application they run exports to the local Grafana instance without configuration. The service name comes from `.qsdev.yaml`'s project name field (Phase 13 Unit 13.1). The `service.version` resource attribute uses the current git SHA for correlation between traces and code versions.

The env vars are injected into devenv.nix's gdev-owned section, following the same pattern as other env vars generated by Phase 3 (database URLs, service addresses). When observability is disabled, no env vars are generated and no container is started.

**Desired Outcome:** When observability is enabled, opening `devenv shell` automatically exports OTEL env vars so that any OTel-instrumented application exports to the local sidecar. Projects without observability enabled are unaffected.

**Steps:**
1. Define the env var generation template in `templates/observability/otel-env.nix.tmpl`:
   ```nix
   # --- gdev:otel-env ---
   # OpenTelemetry configuration — points to local observability sidecar.
   # Any OTel-instrumented application will export to Grafana at localhost:{{.Observability.GrafanaPort}}.
   env.OTEL_EXPORTER_OTLP_ENDPOINT = "http://localhost:{{.Observability.OTLPGRPCPort}}";
   env.OTEL_EXPORTER_OTLP_PROTOCOL = "grpc";
   env.OTEL_SERVICE_NAME = "{{.ProjectName}}";
   env.OTEL_RESOURCE_ATTRIBUTES = "deployment.environment=dev,service.namespace={{.ProjectName}}";
   env.GDEV_GRAFANA_PORT = "{{.Observability.GrafanaPort}}";
   # --- end gdev:otel-env ---
   ```
2. Implement git SHA injection for `service.version`:
   - At generation time, check if `.git` exists.
   - If yes: run `git rev-parse --short HEAD` and embed the result as `service.version=<sha>`.
   - If no (not a git repo): omit `service.version` from `OTEL_RESOURCE_ATTRIBUTES`.
   - The SHA is embedded at generation time, not at shell entry — it reflects the environment at `qsdev init` time, not at each shell open.

### Unit 33.4: Metadata-Only Team Analytics Engine

**Description:** Implement a hook-based event emission engine that collects session metadata — never source code, prompt text, or conversation content. Output is JSONL at `~/.qsdev/analytics/` or a configurable OTel endpoint. Privacy architecture is enforced by schema validation that rejects fields over 256 characters as a content-leaking proxy.

**Context:** The consulting adoption research identified metadata-only team analytics as the biggest unmet consulting need in the Claude Code ecosystem. No existing tool (Rudel, Mantra, ccusage) provides team-level metrics with consulting-appropriate privacy. Rudel uploads full transcripts with no client isolation. Mantra requires closed-source security review. The research proposed a hook-based metadata collection design: PostToolUse and Stop hooks emit structured JSON events containing session ID, project hash (not name), duration, token count, cost estimate, and tool use counts — but nothing from the conversation itself.

The event schema is the privacy boundary. Fields with content limits enforce that no prompt text, code snippets, or file paths leak into the metadata stream. A 256-character field limit is the heuristic proxy: legitimate metadata (session IDs, hash strings, tool names, durations) never approaches this limit; content leaks (prompt text, code, error messages) routinely exceed it. Any field exceeding 256 chars causes the event to be rejected and logged as a privacy violation to `~/.qsdev/analytics/violations.log`.

The engine reads Phase 32's SOC 2 logging hook output as its primary input source — the same structured JSONL events already emitted for compliance logging can be filtered and transformed into analytics events without duplicating hook infrastructure.

**Desired Outcome:** The analytics engine collects session-level metadata after each Claude Code session ends, enforces the privacy schema, and writes JSONL to `~/.qsdev/analytics/`. No content from the conversation is ever stored.

**Steps:**
1. Define the analytics event schema in `internal/analytics/schema.go`:
   ```go
   // AnalyticsEvent is the privacy-safe session metadata event.
   // All string fields are limited to 256 characters. No content fields.
   type AnalyticsEvent struct {
       // Immutable identifiers
       EventType  string `json:"event_type"`  // "session_end", "tool_invocation", "hook_outcome"
       SchemaVer  int    `json:"schema_ver"`   // 1
       Timestamp  string `json:"timestamp"`    // RFC3339

       // Session identifiers (non-reversible project hash, not name)
       SessionID  string `json:"session_id"`   // UUID from Claude Code session
       ProjectHash string `json:"project_hash"` // sha256(project_root)[:16]
       DeveloperID string `json:"developer_id,omitempty"` // opt-in only

       // Quantitative metrics (no content)
       DurationSec    int     `json:"duration_sec,omitempty"`
       InputTokens    int     `json:"input_tokens,omitempty"`
       OutputTokens   int     `json:"output_tokens,omitempty"`
       CostUSD        float64 `json:"cost_usd,omitempty"`
       ToolUseCount   int     `json:"tool_use_count,omitempty"`
       HookBlockCount int     `json:"hook_block_count,omitempty"`

       // Categorical metadata (no content, enumerated values only)
       ExitReason  string            `json:"exit_reason,omitempty"`   // "completed", "interrupted", "error"
       ToolCounts  map[string]int    `json:"tool_counts,omitempty"`   // {"Edit":5,"Bash":12}
       HookOutcomes map[string]int   `json:"hook_outcomes,omitempty"` // {"block":2,"allow":15}
       ModelID     string            `json:"model_id,omitempty"`      // e.g., "claude-sonnet-4-6"
   }

   const MaxFieldLength = 256
   ```
2. Implement the privacy schema validator:
   ```go
   // ValidateEvent enforces the privacy contract: no string field may exceed MaxFieldLength.
   // Returns nil if valid. Returns PrivacyViolation if any string field exceeds the limit.
   func ValidateEvent(evt *AnalyticsEvent) *PrivacyViolation {
       violations := []string{}
       v := reflect.ValueOf(*evt)
       t := v.Type()

       for i := 0; i < v.NumField(); i++ {
           f := v.Field(i)
           if f.Kind() == reflect.String && len(f.String()) > MaxFieldLength {
               violations = append(violations, fmt.Sprintf(
                   "field %s exceeded %d chars (got %d)",
                   t.Field(i).Name, MaxFieldLength, len(f.String()),
               ))
           }
       }

       if len(violations) > 0 {
           return &PrivacyViolation{
               EventType:  evt.EventType,
               Timestamp:  evt.Timestamp,
               Violations: violations,
           }
       }
       return nil
   }
   ```
3. Implement the Stop hook handler that converts a Claude Code session-end event to an analytics event:
   - Hook file: `~/.qsdev/hooks/analytics-stop.sh`
   - Reads the Claude Code Stop hook JSON payload from stdin.
   - Extracts: session ID, model, total tokens (from `usage` field), tool use counts.
   - Computes project hash from `CLAUDE_PROJECT_ROOT` env var: `sha256sum <<< "$CLAUDE_PROJECT_ROOT" | head -c 16`.
   - Constructs `AnalyticsEvent` JSON and appends to `~/.qsdev/analytics/events.jsonl`.
   - Duration: computes from `session_start_time` in the payload vs current time.
4. Implement JSONL output writer:
   ```go
   type AnalyticsWriter struct {
       Path     string
       Rotation RotationPolicy // daily rotation
   }

   func (w *AnalyticsWriter) Write(evt *AnalyticsEvent) error {
       if v := ValidateEvent(evt); v != nil {
           return w.logViolation(v)
       }

       line, err := json.Marshal(evt)
       if err != nil {
           return err
       }

       f, err := os.OpenFile(w.currentPath(), os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0o600)
       if err != nil {
           return err
       }
       defer f.Close()
       _, err = fmt.Fprintf(f, "%s\n", line)
       return err
   }

   func (w *AnalyticsWriter) currentPath() string {
       date := time.Now().Format("2006-01-02")
       return filepath.Join(w.Path, fmt.Sprintf("events-%s.jsonl", date))
   }
   ```
5. Implement configurable OTel output when observability sidecar is running (Unit 33.1):
   - When `OTEL_EXPORTER_OTLP_ENDPOINT` is set, emit analytics events as OTel log records in addition to local JSONL.
   - Use OTel log bridge API, not traces (analytics events are not traces).
   - OTel output is additive — local JSONL is always written regardless of OTel availability.
6. Implement `DeveloperID` opt-in:
   - Default: omit `developer_id` (fully anonymous by project hash only).
   - Opt-in via `~/.qsdev/config.yaml`: `analytics.developer_id: <alias>`.
   - Developer ID is a user-chosen alias string, not email or real name.
   - When set, include in events. When unset, field is absent from all events.
7. Implement `qsdev enable analytics` / `qsdev disable analytics`:

### Unit 33.6: Session Replay Export (Opt-in)

**Description:** Implement `qsdev session export <session-id>` to generate a self-contained HTML replay of a Claude Code session. Opt-in only, never automatic. Credential patterns are redacted before export. Output is a single HTML file viewable in any browser with no external dependencies.

**Context:** Session replay is a knowledge transfer tool for consulting teams: "here's how I debugged the authentication issue on the Jenkins pipeline" can be communicated as a standalone HTML file rather than a verbal explanation. The consulting adoption research ranked session replay (claude-replay, 573 stars) as the second adoption tier after cost visibility tools, specifically for cross-consultant knowledge transfer. The privacy analysis established that session replay is safe for consulting with credential redaction configured — the concern is accidental secret exposure in the JSONL session files. `qsdev session export` bakes credential redaction into the export path so it is not an optional step users can skip.

The approach reads Claude Code's local JSONL session files (at `~/.claude/projects/<project-hash>/`) and renders them as static HTML. No server, no remote calls, no external JavaScript CDN dependencies — the HTML file is self-contained so it can be shared via email or committed to a private repository as documentation.

**Desired Outcome:** `qsdev session export <session-id>` produces a single redacted HTML file that can be shared with another consultant to replay the session. Secret patterns are automatically redacted. The export is explicit — it never happens automatically.

**Steps:**
1. Implement `qsdev session export` command:
   ```go
   var sessionExportCmd = &cobra.Command{
       Use:   "export <session-id>",
       Short: "Export a Claude Code session as a self-contained HTML replay (opt-in)",
       Args:  cobra.ExactArgs(1),
       RunE:  runSessionExport,
   }
   ```
2. Implement session file location:
   - Claude Code stores session JSONL at `~/.claude/projects/<project-hash>/<session-id>.jsonl`.
   - If `<session-id>` is a partial prefix, resolve to the first matching session.
   - `qsdev session list` subcommand: lists available sessions for the current project with date, duration, and first user message (first 80 chars, redacted).
3. Implement credential redaction using the same pattern as Phase 32.2 (secret scanning):
   - Patterns to redact: AWS key IDs (`AKIA[A-Z0-9]{16}`), private key blocks, bearer tokens (`Bearer [A-Za-z0-9._-]{20,}`), generic secrets (`(?i)(secret|password|token|api.key)\s*[:=]\s*[^\s]{8,}`).

   - Redaction applied to all string fields in the JSONL before HTML rendering.

4. Implement HTML renderer:
   - Read JSONL events, classify each as: user message, assistant message, tool use, tool result.
